## Usage

Activate this skill by:
- Using the `/accelint-onboard-openspec` command
- Saying "set up openspec", "onboard to openspec", "generate openspec config", "help me set up openspec", or "I want to use openspec"
- Mentioning "openspec config", "config.yaml for openspec", or "QRSPI config"

Once activated, the skill runs a conversational interview to gather project facts and generates a complete `openspec/config.yaml` file configured for the QRSPI (Question → Research → Design → Structure → Propose → Implement) methodology.

The skill operates in one of three modes based on file state:
- **Create mode** - Full interview from scratch for new projects
- **Import mode** - Merges existing non-standard configs into QRSPI schema
- **Refresh mode** - Abbreviated interview covering only detected drift and unresolved TODOs

## When to Use

Use this skill when:
- Starting a new OpenSpec project and need to create config.yaml
- Converting an existing project to OpenSpec with QRSPI methodology
- Refreshing stale openspec config after significant codebase changes
- Importing and restructuring a custom config.yaml into the standard schema
- Ensuring config.yaml includes all required sections for downstream AI artifacts

## What It Generates

The skill produces a complete `openspec/config.yaml` with:

**Context Section** - Objective project facts injected into every AI artifact:
- Stack facts (runtime, language, frameworks, testing, CI/CD)
- Architecture patterns (organization, path aliases, design patterns)
- Domain concepts (key entities with definitions)
- Performance targets
- Code patterns (exports, naming, error handling, validation)
- Testing patterns (AAA structure, property-based testing, assertions)

**Rules Section** - Per-artifact quality checkpoints:
- Proposal rules (scope definition, boundaries)
- Design rules (current state, desired state, decisions)
- Tasks rules (vertical slicing, granularity, testing gates)
- Spec rules (Given/When/Then, edge cases)

**Anti-Patterns** - Banned patterns across code, performance, testing, and documentation domains

## Key Workflow

### Phase 0: File State Detection

Before asking questions, the skill:
1. Checks for related documentation (ARCHITECTURE.md) to pre-fill infrastructure answers
2. Assesses openspec/config.yaml state:
   - Missing → Create mode (full interview)
   - Empty/near-blank → Create mode with overwrite confirmation
   - Has recognized schema → Refresh mode (drift + TODOs only)
   - Has unrecognized content → Import mode (three options: restructure, append, dry run)

### Phase 1: Discovery Interview

Conversational interview grouped into natural topic turns:
- **Turn 1** - Project identity (name, purpose, monorepo structure, build system)
- **Turn 2** - Tech stack (runtime, language, frameworks, testing, CI/CD)
- **Turn 3** - Architecture (organization, path aliases, design patterns)
- **Turn 4** - Domain concepts (3-5 key entities, specialized terminology)
- **Turn 5** - Performance targets (concrete metrics, hot paths, constraints)
- **Turn 6** - Code patterns (exports, naming, error handling, testing structure)
- **Turn 7** - Anti-patterns (banned patterns, deprecated code, performance traps)
- **Turn 8-10** - Per-artifact rules (proposal, design, tasks)

### Phase 2: Smart Defaults

After each stack answer, surfaces relevant conventions for confirmation:
- Next.js + TypeScript → App Router vs Pages Router, Server Component boundaries
- React + Vitest → userEvent over fireEvent, screen query priority
- Python + FastAPI → Pydantic version, dependency injection patterns
- Node.js + Prisma → transaction patterns, migration conventions

### Phase 3: Parallel Codebase Inference

Spawns 4 parallel discovery subagents to fill remaining gaps:
- **Agent A** - Stack &amp; build tooling (runtime version, TS config, package manager, monorepo workspaces)
- **Agent B** - Testing &amp; code quality (test framework, linters, test structure, type checking config)
- **Agent C** - Architecture &amp; code patterns (directory organization, path aliases, export style, naming conventions)
- **Agent D** - CI/CD &amp; versioning (workflows, release strategy, documented anti-patterns)

Each field gets tagged as `INFERRED [source]` or `UNKNOWN` in the preview. Fields that cannot be inferred are marked `# TODO: fill in`.

### Phase 4: Generation

1. Shows labeled preview with inference sources and TODOs
2. Asks for confirmation and corrections
3. Writes to `openspec/config.yaml` (strips inference comments from final file)
4. Validates YAML syntax (no tabs, proper quoting, no syntax errors)
5. Prints summary of configured vs inferred vs TODO fields

## Common Pitfalls

> **Good to know:** This skill focuses on **project DNA** (WHAT the project is), not agent behavior (HOW agents act). Commit conventions, workflow procedures, and tool preferences belong in AGENTS.md via the `accelint-onboard-agents` skill.

> **Good to know:** Parallel inference in Phase 3 is mandatory — spawning 4 discovery agents simultaneously is faster than serial scanning, especially for monorepos with many config files.

> **Good to know:** Fields marked `# TODO: fill in` are intentional — they preserve config structure and prompt action. Silent omission creates invisible gaps that degrade all downstream AI artifacts.

> **Good to know:** YAML is strict about special characters. Values starting with `(`, `[`, `|`, `"` must be quoted. The skill validates syntax after generation and fixes issues automatically.

## YAML Generation Safety

The skill follows strict quoting rules to avoid syntax errors:

**Special characters requiring quotes:** `|`, `>`, `"`, `'`, `(`, `)`, `[`, `]`, `{`, `}`, `*`, `&`, `!`, `%`, `@`, `` ` ``

**Examples:**
```yaml
# Correct quoting
description: "(internal) auth module"
tag: "[PKG:auth]"
pattern: "some|other"
command: 'npm run "test:unit"'

# Multi-line content uses block scalars
context: |
  Line 1
  Line 2
```

**Post-generation validation checks:**
- No tab characters (YAML requires spaces)
- Consistent 2-space indentation
- Quoted values for special characters
- Valid YAML structure (no unmatched brackets/quotes)
