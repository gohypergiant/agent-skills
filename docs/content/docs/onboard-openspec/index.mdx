## What It Helps You Do

Use this skill to set up OpenSpec for your project through a guided interview that produces a complete `openspec/config.yaml` file configured for the QRSPI methodology.

The skill is especially useful when you need to:
- Generate project-specific OpenSpec configuration from scratch
- Import and restructure existing config files into the QRSPI schema
- Refresh stale configuration when your codebase has evolved
- Document your project's architectural facts and patterns for AI-generated artifacts

Activate this skill by:
- Using the `/accelint-onboard-openspec` command
- Saying "set up openspec", "onboard to openspec", or "generate openspec config"
- Mentioning "QRSPI config" or "config.yaml for openspec"

## When to Use

Use this skill when:
- Users mention "openspec config", "config.yaml for openspec", or "QRSPI config"
- Starting a new project that will use OpenSpec for spec-driven development
- You have an existing config that needs to be restructured or updated
- Your codebase has changed significantly since the config was last generated

## How It Works

The skill runs a conversational interview across multiple phases to build your configuration:

**Discovery Interview** — Asks targeted questions about your project in grouped topics:
- Project identity (name, purpose, monorepo structure)
- Tech stack (runtime, language, frameworks, libraries, tooling)
- Architecture patterns (organization, path aliases, design patterns)
- Domain concepts (key entities and terminology)
- Performance targets and constraints
- Code patterns (naming, exports, error handling, testing)
- Anti-patterns (banned or deprecated approaches)
- Per-artifact rules (proposal, design, tasks, spec requirements)

**Smart Defaults** — After each stack answer, surfaces relevant conventions to confirm instead of asking every detail. For example, detecting Next.js prompts confirmation about App Router vs Pages Router patterns.

**Parallel Codebase Inference** — Spawns four simultaneous discovery agents to fill gaps by scanning your codebase:
- Agent A: Stack and build tooling (runtime versions, TypeScript config, package manager)
- Agent B: Testing and code quality (test framework, linters, test structure patterns)
- Agent C: Architecture and code patterns (directory organization, naming conventions, error handling)
- Agent D: CI/CD and versioning (workflows, release strategy, documented anti-patterns)

Each inferred value is tagged with its source for transparency. Values that cannot be resolved are marked with `# TODO: fill in` so you know what needs manual input.

**Preview and Validation** — Shows the complete generated config with labeled inference sources before writing anything. After you confirm, the skill validates the YAML syntax to catch any formatting issues.

### Intelligent Mode Detection

Before asking any questions, the skill checks whether `openspec/config.yaml` exists and adapts its approach:

**Create Mode** — File doesn't exist or is empty. Runs the full interview from scratch.

**Import Mode** — File has unrecognized content. Presents three options:
- **(a) Restructure** — Maps existing content onto the QRSPI schema, flags behavioral content that belongs in AGENTS.md instead, runs targeted interview for gaps
- **(b) Append** — Adds QRSPI sections alongside existing content without modifying what's there
- **(c) Dry run** — Shows what would be generated without changing any files

**Refresh Mode** — File matches the expected schema. Runs abbreviated interview covering only:
- Drift detection (scanning for changes since last update: new packages, tooling changes, config tightening)
- Unresolved TODOs (surfaces `# TODO: fill in` markers from previous run)
- External findings (if the prompt includes a `findings:` list from completed changes)

## Good to Know

> **Good to know:** This skill produces the **project DNA layer** (structural facts about what the project is). It is the companion to `accelint-onboard-agents`, which produces the **behavior layer** (how the agent acts and makes decisions). If you volunteer behavioral content during the interview—commit conventions, workflow steps, tool preferences—the skill will note it for AGENTS.md instead of adding it to config.yaml.

> **Good to know:** The skill attempts codebase inference before asking questions. If inference fails for a field, it marks the section with `# TODO: fill in` rather than omitting it. An explicit TODO is actionable; a missing section silently degrades every AI artifact it drives.

> **Good to know:** YAML syntax is strict about special characters. The skill automatically quotes values that start with special characters like `(`, `[`, `|`, or `:` to prevent syntax errors. After writing, it validates the generated YAML and fixes any issues immediately.

## Prerequisites

- Basic understanding of your project's tech stack and architecture
- Access to the project repository
- (Optional) ARCHITECTURE.md file — if present, the skill reads it to avoid asking questions about deployment and infrastructure that are already documented

## What You Get

A complete `openspec/config.yaml` file with two main sections:

**context:** — Objective facts about your codebase injected into every AI-generated artifact (proposal, design, spec, tasks). Includes:
- Stack facts (runtime, language, frameworks, tooling)
- Architecture patterns (organization, path aliases, design patterns)
- Domain concepts (key entities and terminology)
- Performance targets
- Code patterns to follow
- Anti-patterns to avoid

**rules:** — Per-artifact quality checkpoints that encode your team's standards:
- Proposal rules (scope definition requirements, length limits)
- Design rules (required sections, diagram preferences, technical depth)
- Task rules (vertical slicing preferences, granularity, test requirements)
- Spec rules (Given/When/Then format, edge case documentation)

## Examples

### Example: Creating config from scratch

```bash
# Start the onboarding interview
/accelint-onboard-openspec

# The skill detects no existing config and runs Create Mode
# After answering interview questions and reviewing the preview:
# Output: openspec/config.yaml created with complete QRSPI configuration
```

### Example: Refreshing stale config

```bash
# Run the skill when your codebase has evolved
/accelint-onboard-openspec

# The skill detects existing config matching QRSPI schema
# Runs Refresh Mode: scans for drift, surfaces TODOs
# Shows only changed sections in preview before updating
```

### Example: Importing existing config

```bash
# You have an existing config with custom structure
/accelint-onboard-openspec

# The skill detects unrecognized content
# Presents options: (a) Restructure, (b) Append, (c) Dry run
# Choose (a) to map existing content onto QRSPI schema
# Shows merged preview with labels: "# from existing file" vs "# new"
```
