## What It Helps You Do

Use this skill to audit your OpenSpec archive for long-term inconsistencies that individual change workflows do not catch.

It is especially useful when you need to find:
- **Decision drift** between archived changes that no longer agree
- **Stale index rows** where `specs/INDEX.md` no longer matches the underlying spec files
- **Structural over-coupling** where a capability now touches too many related specs

Unlike the rest of the QRSPI/OpenSpec workflow, this skill looks backward across the full archive instead of evaluating one change at a time.

Activate by saying "run archive synthesis," "lint the openspec archive," or "check for decision drift."

## When to Use

Use this skill when:

- `accelint-qrspi-archive` suggests running synthesis (typically after 15+ changes)
- Checking whether an old design decision still holds given everything decided since
- Verifying `specs/INDEX.md` still matches the actual spec.md files on disk
- Finding capabilities that have become over-coupled refactor candidates
- Auditing the archive for contradictions before a major milestone or release

## How It Works

The skill runs three archive-wide checks:

### Decision Drift

It looks for archived changes that appear to make conflicting decisions about the same or related capabilities, then verifies only the flagged candidates in more detail.

### Index Reconciliation

It checks whether `specs/INDEX.md` still matches the actual spec files on disk, including basic existence, purpose text, and `related:` metadata.

### Structural Coupling

It flags capabilities whose relationship count suggests they may have grown beyond their intended boundaries and should be reviewed as technical debt.

## What You Get

A structured report in the same CRITICAL/WARNING/SUGGESTION format as `/opsx:verify`:

```
## Archive Synthesis Report — 2026-07-06
Corpus: 47 archived changes, 22 capabilities, checked back to 2026-01-10.

### CRITICAL
- [Confirmed decision contradiction with actively-touched capability]
- [Missing capability directory or spec.md file]

### WARNING
- [Confirmed decision contradiction on dormant capability]
- [Index row disagrees with its own spec.md]

### SUGGESTION
- [Staleness flag]
- [Structural coupling flag]
```

For each finding, you choose:

- **Confirm** — the skill updates `Status` (for decision drift), patches/removes the index row (for reconciliation), or routes to the relevant doc skill via the shared `findings:` interface
- **Dismiss** — for decision drift, recorded permanently; for coupling/reconciliation, not persisted (resurfaces next run if still present)
- **Defer** — left as-is; resurfaces next run

## Prerequisites

- OpenSpec CLI installed and initialized
- `accelint-qrspi-archive` already in regular use (both indexes must exist with at least one row each)
- Sub-agent support for targeted `design.md` and `spec.md` reads
- Read access to every capability directory under `openspec/specs/`
- For automatic routing to work, writer skills (`accelint-architecture-doc`, `accelint-onboard-openspec`, `accelint-onboard-agent`, `accelint-readme-writer`) need Mode 3 Refresh support with the `findings:` interface

> **Good to know:** If writer skills don't support `findings:` yet, the skill still produces the report — you just handle the routing manually.

## Good to Know

> **Good to know:** This skill never runs automatically and does not block other QRSPI/OpenSpec workflows. It is an explicit audit step.

> **Good to know:** Confirmed index reconciliation findings can patch `specs/INDEX.md`, but the skill does not rewrite spec files themselves.

> **Good to know:** Dismissed decision-drift pairs are recorded in `openspec/changes/archive/SYNTHESIS-LOG.md` so the same dismissed contradiction does not keep resurfacing.

> **Good to know:** Thresholds such as the 15-change trigger and the structural-coupling cutoff are defaults for review, not automatic policy changes.

## Limits

This skill does not:

- Run automatically or block any other skill's workflow
- Edit hub docs directly (routes through writer skills via `findings:`)
- Write to `archive/INDEX.md` except the `Status` column for confirmed decision drift
- Write to `specs/INDEX.md` except patching/removing a single confirmed reconciliation row
- Touch any `design.md` or `spec.md` file (reads only)
- Operate across repositories (local project only)
- Adjust thresholds on its own authority

## Examples

### Example: Suggested run after 15 changes

```bash
# accelint-qrspi-archive suggests synthesis
/accelint-archive-synthesis

# Output:
# Preflight checks pass
# Scans archive, finds 1 CRITICAL decision drift
# Human confirms, names which change stands
# Updates Status, routes to accelint-architecture-doc
# Logs checkpoint to SYNTHESIS-LOG.md
```

### Example: First-ever run on a small corpus

```bash
/accelint-archive-synthesis

# Output:
# ⚠ 6 archived changes is a small corpus
# Continue anyway? Yes
# No findings (too few changes for meaningful signal)
# Creates SYNTHESIS-LOG.md fresh
```

### Example: Dismissed pair stays suppressed

```bash
# Run 1: human dismisses a decision-drift finding
# Skill records to SYNTHESIS-LOG.md dismissed: list

# Run 2 (months later):
/accelint-archive-synthesis

# Output:
# Loads dismissed-pair history
# Skips that specific pair without re-verification
# Reports "previously dismissed — not re-surfaced this run"
```
