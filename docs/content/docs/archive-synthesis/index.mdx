## Usage

Activate this skill by:
- Using the `/accelint-archive-synthesis` command
- Saying "run archive synthesis", "lint the openspec archive", "check for decision drift"
- Asking to "audit the spec archive for contradictions", "find stale specs", "check capability coupling"
- Mentioning "reconcile the specs index" or "check if old design decisions still hold"

```bash
# Typically invoked when accelint-qrspi-archive suggests it after ~15 archived changes
/accelint-archive-synthesis

# Or manually at any time
/accelint-archive-synthesis
```

Once activated, the skill reads backward across the entire archived-change history to check whether it still agrees with itself. It scans two indexes (`openspec/changes/archive/INDEX.md` and `openspec/specs/INDEX.md`), performs targeted verification of flagged candidates, and produces a report with CRITICAL/WARNING/SUGGESTION findings in the same register as `/opsx:verify`.

> **Good to know:** This skill never runs automatically and never blocks another skill's workflow—it's always either a human-invoked audit or an offered suggestion the human accepts explicitly.

> **Good to know:** For confirmed decision-drift findings, you'll be asked to specify which side of the contradiction stands before any Status update is written—the skill never infers this from recency alone.

## When to Use

Use this skill when:
- Users say "run archive synthesis", "lint the openspec archive", "check for decision drift"
- `accelint-qrspi-archive` suggests that 15+ changes have archived since the last synthesis run
- You want to audit the spec archive for contradictions or inconsistencies
- Investigating whether an old design decision still holds given everything decided since
- Checking whether `specs/INDEX.md` still matches the actual `spec.md` files on disk
- Determining if some capability has become an over-coupled refactor candidate
- Running a corpus health check manually

## What It Checks

**Decision Drift** - Archived decisions that no longer cohere with each other (e.g., one change chose polling due to budget constraints, later change adopted a message broker for unrelated capability)

**Index Reconciliation** - `specs/INDEX.md` rows that disagree with actual `spec.md` files or point to directories/files that no longer exist

**Structural Coupling** - Capabilities whose `related:` count indicates they've outgrown their boundaries (≥5 relationships and ≥2× the index median)

**Staleness Signals** - Capabilities with old `Last touched by` dates relative to clustered recent activity in related capabilities

## Key Patterns

### The Three Checks

**Decision Drift Detection:**
1. Group archived changes by shared or related capabilities
2. Coarse-scan `Decision` column for opposing patterns (polling vs push, eager vs lazy, sync vs async, etc.)
3. Spawn subagents to verify flagged candidates against full `design.md` files
4. Classify as CRITICAL if capability was touched after earlier change, WARNING otherwise

**Index Reconciliation:**
1. Verify every `specs/INDEX.md` row's capability directory and `spec.md` file exists (CRITICAL if missing)
2. Read `## Purpose` heading and `related:` frontmatter from each `spec.md`
3. Compare against index row values (WARNING if mismatched)

**Structural Coupling:**
1. Compute median `related:` list length across all capabilities
2. Flag any with count ≥5 AND ≥2× median (SUGGESTION level)
3. Report as plain fact, route to ARCHITECTURE.md's Known Technical Debt section

### Cost Control at Scale

The skill achieves efficient scaling through:

- **Index-first scanning** - Reads structured indexes (stay cheap regardless of history length), not raw files
- **Targeted verification** - Opens `design.md` only for specific candidates the coarse scan flagged
- **Subagent delegation** - Keeps raw design content out of parent context on every run
- **Dismissed-pair tracking** - Skips pairs already ruled out permanently in prior runs

### Human Review and Routing

Every finding offers three options:

**Decision-drift findings:**
- **Confirm** - Skill asks which change stands, updates `Status` column, routes to relevant writer skill(s)
- **Dismiss** - Recorded permanently to `SYNTHESIS-LOG.md`; pair never flagged again
- **Defer** - Left as-is; resurfaces next run

**Structural coupling/reconciliation findings:**
- **Confirm** - Routes to appropriate writer skill or patches index row directly
- **Dismiss** - NOT persisted; finding resurfaces next run (live snapshot, not historical fact)
- **Defer** - Left as-is; resurfaces next run

> **Good to know:** Only decision-drift dismissals persist. Structural coupling and index reconciliation are live snapshots that deserve re-checking every run regardless of prior dismissals.

### The Two Indexes (Owned Elsewhere)

This skill reads but doesn't write these indexes (maintained by `accelint-qrspi-archive`):

```markdown
openspec/changes/archive/INDEX.md — one row per archived change:
| Change        | Date       | Decision                                  | Specs touched                      | Status  |
|---------------|------------|-------------------------------------------|------------------------------------|---------|
| add-live-sync | 2026-03-02 | polling with 5s interval, no infra budget | sync/protocol, ui/status-indicator | current |
```

```markdown
openspec/specs/INDEX.md — one row per capability:
| Capability    | Purpose                                     | Related             | Last touched by            |
|---------------|---------------------------------------------|---------------------|----------------------------|
| sync/protocol | Defines how client and server exchange state | ui/status-indicator | add-live-sync (2026-03-02) |
```

Exception: The skill can update a single row's `Status` column in `archive/INDEX.md` after confirmed decision-drift, and can patch or remove a single row in `specs/INDEX.md` after confirmed reconciliation finding.

### The Log (Owned by This Skill)

`openspec/changes/archive/SYNTHESIS-LOG.md` tracks synthesis checkpoints:

```markdown
2026-07-06 — checked through row 42 — 2 confirmed, 1 dismissed, 1 deferred
  dismissed: [add-live-sync|adopt-notification-gateway|sync/protocol]
```

This append-only log enables:
- Computing "changes since last run" for cadence sanity-checks
- Permanently suppressing dismissed decision-drift pairs across all future runs

## Common Pitfalls

> **Good to know:** This skill focuses on backward-looking consistency across the archive. For forward-looking checks (does implementation match design.md?), use `/opsx:verify` before archiving. The two are complementary—a change can pass verification cleanly and still surface in a decision-drift finding years later.

> **Good to know:** Dismissing a structural coupling or index reconciliation finding does NOT permanently suppress it—both are live snapshots that get re-checked every run. Only decision-drift dismissals persist to the log, since those are anchored to immutable archived changes.

> **Good to know:** The skill can detect contradictions but never resolves them on its own authority. After confirming a decision-drift finding, you'll always be asked which side stands before any Status update is written.

> **Good to know:** This skill routes findings to writer skills via the shared `findings:` interface—it never rewrites hub docs directly. If a writer skill doesn't support the interface yet, the finding is presented in the report but needs manual follow-up.

> **Good to know:** The 15-change trigger and 5-and-2×-median coupling thresholds are reasoned defaults, not measured ones. Preflight checks report mismatches but never adjust automatically—changing numbers is a human's call based on your project's actual velocity and relationship density.
