## What It Helps You Do

Use this skill to systematically improve documentation across JavaScript and TypeScript codebases. It helps you add comprehensive JSDoc comments to public APIs, clean up comment clutter, and apply proper comment markers for actionable issues.

The skill applies intelligent judgment about what to document based on visibility:
- **Exported code** gets comprehensive documentation (consumers need full context)
- **Internal code** gets minimal documentation (team members can read implementation)

This prevents documentation rot while ensuring public APIs remain well-documented.

## When to Use

Use this skill when:
- Adding JSDoc comments (@param, @returns, @template, @example)
- Auditing documentation completeness
- Improving comment quality and removing comment clutter
- Adding proper comment markers (TODO, FIXME, HACK, NOTE)
- Validating that exported functions and types are properly documented
- Reviewing code for missing or incomplete documentation

Activate by saying "add JSDoc", "document this function", "audit documentation", "fix comments", "add TODO/FIXME markers", or "improve code documentation".

## How It Works

The skill applies a two-tier documentation strategy:

**Tier 1 - Exported Code (Public API):**
- Requires comprehensive documentation for all exports
- All @param tags with property documentation for objects
- @returns describing outcomes in different scenarios
- @template with constraint explanations for generics
- @throws documenting all possible errors with triggering conditions
- At least one realistic @example showing usage

**Tier 2 - Internal Code:**
- Minimal documentation focused on non-obvious behavior
- Brief descriptions (one line acceptable)
- @param only for non-obvious parameters
- @example only for complex behavior
- Emphasis on WHAT/WHY over HOW

### Expert Judgment Framework

Before documenting, the skill considers:
- **Who is the reader?** API consumers lack implementation context, team members have it
- **Opacity vs complexity?** Document hidden intent, not intricate implementation
- **Maintenance cost?** High-churn code gets minimal docs, stable APIs get comprehensive docs

### JSDoc Standards

When adding JSDoc, the skill ensures:
- Object parameters use dot notation (`@param options.timeout`)
- @example tags use proper code fences with language identifiers
- Descriptions focus on WHAT/WHY, not HOW (implementation details)
- No @returns on void functions
- Generic functions have @template for each type parameter

### Comment Quality

For comment audits, the skill:
- Categorizes comments with proper markers (TODO, FIXME, HACK, NOTE, PERF, REVIEW)
- Removes unnecessary comments (commented-out code, edit history, obvious statements)
- Preserves important comments (markers with context, linter directives, business logic)
- Ensures markers are specific with ownership: `TODO(username): Replace with binary search`

### Audit Reports

When explicitly invoked for audits, the skill produces structured reports with:
- Numbered findings with before/after examples
- Categorization (Missing, Incomplete, Incorrect Syntax, Quality)
- References to detailed guidance
- Summary table for tracking issues

## Examples

### Example: Adding Comprehensive Public API Documentation

For exported functions, the skill adds complete JSDoc:

```typescript
/**
 * Fetches user profile data from the authentication service
 *
 * Automatically retries up to 3 times on network failures with exponential
 * backoff. Throws if user is not authenticated or profile doesn't exist.
 *
 * @param userId - Unique identifier for the user profile to fetch
 * @param options - Configuration for fetch behavior
 * @param options.includeMetadata - Include account metadata (creation date, last login)
 * @param options.timeout - Request timeout in milliseconds (default: 5000)
 * @returns User profile with email, name, and optional metadata
 * @throws {AuthenticationError} When user session is expired or invalid
 * @throws {NotFoundError} When user profile doesn't exist
 * @throws {NetworkError} When all retry attempts are exhausted
 *
 * @example
 * ```typescript
 * const profile = await fetchUserProfile('user-123');
 * console.log(profile.email);
 * ```
 */
export async function fetchUserProfile(
  userId: string,
  options?: { includeMetadata?: boolean; timeout?: number }
): Promise<UserProfile> {
  // implementation
}
```

### Example: Minimal Internal Documentation

For internal utilities, brief documentation is sufficient:

```typescript
/** Checks if value is not null/undefined */
function isValid(x: unknown): boolean {
  return x != null;
}
```

### Example: Improving Comment Markers

Before:

// TODO(username): Replace with binary search for O(log n) lookup

```

## Good to Know

> **Good to know:** This skill focuses on documentation quality and completeness. For general code quality improvements, use `accelint-ts-best-practices`. For performance optimization, use `accelint-ts-performance`.

> **Good to know:** When performing audits on multiple files, the skill produces a structured report. For single-function documentation requests, it applies changes directly without formal reporting.

> **Good to know:** The skill loads detailed reference files only when needed - `jsdoc.md` for JSDoc tasks, `comments.md` for comment quality tasks. This keeps responses focused and efficient.

## What It Does Not Do

- General code quality improvements (type safety, control flow)
- Performance optimization
- Framework-specific documentation (React PropTypes, Vue props)
- Documentation for test files or mock utilities
