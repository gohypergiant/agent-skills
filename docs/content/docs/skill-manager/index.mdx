## Usage

Activate this skill by:
- Using the `/accelint-skill-manager` command
- Saying "create a skill", "make a new skill", "build a skill", "skill for X"
- Saying "audit this skill", "review this skill", "check skill quality"
- Saying "fix this skill", "improve this skill", "refactor this skill", "update this skill", "optimize this skill"

The skill provides structured workflows:

**For creating new skills:**
```bash
# Command
/accelint-skill-manager

# Follow the 4-step workflow:
# 1. Understanding - Gather concrete examples
# 2. Planning - Identify scripts, references, assets
# 3. Initializing - Create directory structure from template
# 4. Editing - Write agent-focused content with CHANGELOG
```

**For auditing existing skills:**
```bash
# Command
/accelint-skill-manager path/to/skill/

# Runs structured audit:
# - Frontmatter validation
# - Structure audit
# - CHANGELOG and version audit
# - Knowledge delta test
# - Actionable improvement recommendations
```

Once activated, the skill guides you through skill creation or audit with progressive disclosure—loading detailed reference files only when needed to minimize context usage.

## When to Use

Use this skill when:
- Creating a new agent skill from scratch
- Refactoring or improving an existing skill
- Auditing skill quality against specifications
- Unsure how to structure skill content
- Need guidance on freedom calibration (prescriptive vs principle-based)
- Need to validate frontmatter, descriptions, or CHANGELOG format

## Key Concepts

### Progressive Disclosure

The skill uses a three-layer approach to minimize context usage:

**Layer 1: Metadata (~100 tokens)** - Always loaded via description field  
**Layer 2: SKILL.md (&lt;500 lines)** - Core workflow loaded on activation  
**Layer 3: Resources (on-demand)** - References, scripts, assets loaded only when needed

References available:
- `references/file-system.md` - Directory structure and naming
- `references/skill.md` - Description field conventions
- `references/agents.md` - AGENTS.md patterns
- `references/progressive-disclosure.md` - Context optimization rules
- `references/references.md` - Reference file format
- `references/scripts.md` - Script conventions
- `references/assets.md` - Asset guidelines
- `references/changelog.md` - CHANGELOG maintenance

### Freedom Calibration

Match guidance style to task fragility:

| Task Type | Freedom | Format | Example |
|-----------|---------|--------|---------|
| **Creative/Design** | High | Principles, anti-patterns | "Commit to a bold aesthetic" |
| **Code Review** | Medium | Guidelines with examples | "Priority: security > logic > performance" |
| **File Operations** | Low | Exact scripts, specific steps | "Use exact command: `pandoc --flag`" |

**The test:** "If the agent makes a mistake, what's the consequence?"
- High consequence (file corruption, data loss) → Low freedom with precise scripts
- Medium consequence (suboptimal code, style issues) → Medium freedom with examples
- Low consequence (aesthetic choices) → High freedom with principles

### Knowledge Delta Test

For each content block, ask: **"Does Claude already know this?"**

✅ **Expert-only knowledge** - Hard-won patterns, domain anti-patterns, non-obvious rules  
❌ **Redundant knowledge** - Tutorials, standard library usage, generic advice

If &gt;50% redundant, substantial revision is needed.

## Workflows

### Which Workflow to Follow

**Creating a new skill from scratch** → Follow Skill Creation Workflow (4 steps)  
**Improving an existing skill** → Jump to Step 4 (Edit the Skill)  
**Auditing a skill for quality** → Follow Skill Audit Workflow

### Skill Creation Workflow

Track progress with this checklist:

```
- [ ] Step 1: Understanding - Gather concrete examples of skill usage
- [ ] Step 2: Planning - Identify reusable scripts, references, assets
- [ ] Step 3: Initializing - Check existing skills, create directory structure
- [ ] Step 4: Editing - Write agent-focused content and update CHANGELOG
```

#### Step 1: Understanding with Concrete Examples

Ask 2-3 concrete questions to understand skill usage:
- What functionality?
- Usage examples?
- Trigger phrases?

Example for `image-editor` skill:
- "What functionality? Editing, rotating, other?"
- "Usage examples?"
- "Trigger phrases: 'Remove red-eye', 'Rotate image'—others?"

Follow up based on answers rather than front-loading all questions.

#### Step 2: Planning Reusable Contents

Analyze each example to identify reusable resources:

**Scripts** - Avoid rewriting code each time  
Example: `pdf-editor` stores `scripts/rotate-pdf.sh`

**Assets** - Boilerplate templates  
Example: `frontend-app-builder` stores `assets/hello-world/` template

**References** - Domain knowledge loaded on-demand  
Example: `big-query` stores `references/schema.md` with table schemas

#### Step 3: Initializing the Skill

**Load `references/file-system.md` before creating directory structure.**

For new skills, copy the template:
```bash
cp -r skills/accelint-skill-manager/assets/skill-template/ skills/new-skill-name/
```

Before creating, check for existing skills:
```bash
ls -la .claude/skills 2>/dev/null || echo "No project skills"
ls -la ~/.claude/skills 2>/dev/null || echo "No global skills"
```

#### Step 4: Edit the Skill

**Load `references/skill.md` for description field conventions.**

Focus on procedural knowledge that helps another agent instance execute tasks effectively. Consider:
- Non-obvious decisions
- Domain-specific workflows
- Edge cases Claude wouldn't know

**Version Control:**
- **Major (1.0 → 2.0):** Substantial rewrites, breaking changes
- **Minor (1.0 → 1.1):** New sections, significant additions
- **Patch (1.0.0 → 1.0.1):** Bug fixes, typo corrections (optional third digit)

**CHANGELOG Maintenance:**

Update `CHANGELOG.md` after changes using "Keep a Changelog" format:

```markdown
# Changelog

## [X.Y.Z] - YYYY-MM-DD

### Added
- New features with rationale

### Changed
- Modifications with why (rationale is critical)

### Fixed
- Bug fixes with explanation

### Version
- Version bump note
```

The CHANGELOG version must match frontmatter `metadata.version`.

### Skill Audit Workflow

#### 1. Frontmatter Audit

Check each field:
- `name`: lowercase, hyphens only, ≤64 chars, matches directory name
- `description`: starts with "Use when", includes WHAT/WHEN/KEYWORDS, concrete trigger phrases
- `license`: present (optional but recommended)
- `metadata.version`: present, matches CHANGELOG.md latest version

#### 2. Structure Audit

Compare against expected sections:
- NEVER Do
- Before [Action] Ask
- How to Use
- Main Workflow

Note missing sections.

#### 3. CHANGELOG and Version Audit

Check CHANGELOG.md:
- **Missing CHANGELOG:** Flag as missing documentation
- **Version mismatch:** Verify CHANGELOG matches frontmatter version
- **Missing rationale:** Ensure changes include WHY, not just WHAT
- **Format compliance:** Check "Keep a Changelog" structure
- **Empty sections:** Verify meaningful content

#### 4. Knowledge Delta Test

For each content block, classify as REDUNDANT or EXPERT-ONLY.

Calculate percentage of redundant content. If &gt;50%, recommend substantial revision.

#### 5. Produce Actionable Output

- Specific improvement recommendations ranked by priority
- Concrete improved description (not just criticism)
- Improved SKILL.md alongside audit report

## Anti-Patterns

### NEVER write tutorials explaining basics
Assume Claude knows standard concepts, libraries, and patterns. A skill explaining how to use `Array.map()` wastes tokens on knowledge Claude already has. Focus on expert-only knowledge.

### NEVER put triggering information in body
"When to use" guidance belongs ONLY in the description field. The body is loaded after activation decision, so including triggering conditions there wastes context and causes confusion.

❌ Bad:
```markdown
## When to Use This Skill
Use this skill when working with React components...
```

✅ Good (in frontmatter description only):
```yaml
description: Use when writing React components, optimizing re-renders, fixing hydration...
```

### NEVER dump everything in SKILL.md
Use progressive disclosure. Keep SKILL.md under 500 lines with core workflow. Move detailed content to `references/` for on-demand loading.

❌ Bad: 1200-line SKILL.md with every edge case  
✅ Good: 400-line SKILL.md, 6 reference files loaded as needed

### NEVER use generic warnings
"Be careful" and "avoid errors" provide no actionable guidance. Provide specific anti-patterns with concrete reasons from experience.

❌ Bad: "NEVER skip validation — causes errors"  
✅ Good: "NEVER skip validation of user-provided file paths — leads to directory traversal vulnerabilities where attackers can read arbitrary files by injecting '../' sequences"

### NEVER use same freedom level for all tasks
Creative domains need high freedom with principles. Fragile operations need low freedom with exact scripts. Mismatched freedom causes either over-constraint (stifling creativity) or under-constraint (causing errors).

❌ Bad: Giving exact scripts for UI design (over-constrains creativity)  
❌ Bad: Giving vague principles for file format parsing (under-constrains, causes bugs)  
✅ Good: Principles for design, scripts for parsing

### NEVER skip the anti-patterns section
A skill without "NEVER Do" is missing half of expert knowledge. Anti-patterns capture mistakes experts learned the hard way—the non-obvious failures that take years to discover.

### NEVER write a vague description
"A skill for X" causes false positives and missed activations. The description must answer WHAT, WHEN, and include concrete KEYWORDS users actually say.

❌ Bad: "Helps with React development"  
✅ Good: "Use when writing React components, hooks, JSX; optimizing re-renders; debugging infinite loops; fixing hydration mismatches. Triggers on useEffect, useState, memo..."

## Common Pitfalls

> **Good to know:** Creating a skill vs auditing a skill are distinct workflows. Don't mix creation concerns (gathering examples, planning structure) with audit concerns (validating existing content). Use the "Which Workflow to Follow" section to route to the right process.

> **Good to know:** The template in `assets/skill-template/` includes detailed comments explaining WHY each section matters. Read the comments before deleting them—they provide context on the reasoning behind the structure.

> **Good to know:** When updating an existing skill, bump `metadata.version` in frontmatter AND update `CHANGELOG.md` with the same version number. Version mismatches cause confusion about which iteration is documented.

> **Good to know:** Load reference files only when you need implementation details for that specific rule. Loading all references at once wastes context. The AGENTS.md provides rule summaries—start there and drill down as needed.
