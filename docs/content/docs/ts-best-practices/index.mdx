## What It Helps You Do

Use this skill to write safer, more maintainable TypeScript and JavaScript code. It focuses on type safety, defensive programming, and code correctness — helping you avoid common pitfalls that lead to runtime errors, type confusion, and fragile codebases.

This skill is especially useful when you need to:
- Write or review TypeScript/JavaScript code for quality and safety
- Fix type errors or eliminate `any`, `enum`, or `null` usage
- Implement robust control flow, state management, or error handling
- Apply zero-value patterns or immutability principles
- Conduct code reviews for TypeScript anti-patterns

Activate this skill by saying phrases like "review this code", "fix type errors", "apply best practices", "audit code quality", or when working with `.ts` or `.js` files.

## When to Use

Use this skill when:

## How It Works

This skill uses **progressive disclosure** to minimize token usage while providing comprehensive guidance:

1. **Start with the overview** — Scan rule summaries in `AGENTS.md` to identify relevant patterns
2. **Load specific rules as needed** — Each reference file is self-contained with ❌/✅ examples
3. **Apply the pattern** — Follow the detailed guidance for the specific issue you're addressing

The skill covers critical anti-patterns that frequently appear in codebases:

- **Type Safety** — Never use `any` (use `unknown` or generics), never use `enum` (use `as const` objects), prefer `type` over `interface`
- **Return Values** — Return zero values ([], {}, 0, '') instead of null/undefined to eliminate defensive null checks
- **State Management** — Use `const` to signal immutability, never mutate function parameters, keep leaf functions pure
- **Safety** — Always validate external data with schemas, set explicit limits on loops and queues, handle all errors explicitly
- **Code Quality** — Use descriptive names with qualifier ordering, keep functions under 50 lines, use early returns for guard clauses

When invoked directly or for comprehensive audits, the skill generates a structured report using a standardized template with severity levels (Critical, High, Medium, Low), impact analysis, and pattern references.

### Key Rule Categories

**General Best Practices:**
- Naming conventions (qualifier ordering, boolean prefixes)
- Function design (size, parameters, explicit return types)
- Control flow (early returns, block style)
- State management (const vs let, immutability)
- Return values (zero values instead of null)
- Code duplication (DRY principle)

**TypeScript Patterns:**
- Avoiding `any` type
- Using `as const` objects instead of `enum`
- Preferring `type` over `interface`
- Using statically analyzable import paths

**Safety:**
- Input validation at system boundaries
- Assertions with included values
- Explicit error handling
- Bounded iteration to prevent resource exhaustion

## Examples

### Example: Eliminating `any` Types

```typescript
// ❌ Before: any disables type checking
function processData(input: any): any {
  return input.value;
}

// ✅ After: unknown forces validation
function processData(input: unknown): string {
  if (typeof input === 'object' && input !== null && 'value' in input) {
    return String(input.value);
  }
  throw new Error('Invalid input structure');
}
```

### Example: Zero-Value Returns

```typescript
// ❌ Before: null checks everywhere
function getUsers(): User[] | null {
  if (!hasUsers) return null;
  return users;
}
const result = getUsers();
if (result !== null) {
  result.forEach(u => console.log(u));
}

// ✅ After: no defensive checks needed
function getUsers(): User[] {
  if (!hasUsers) return [];
  return users;
}
getUsers().forEach(u => console.log(u));
```

### Example: Replacing Enums

```typescript
// ❌ Before: enum generates 5+ lines of runtime code
enum Status {
  Active = 'active',
  Inactive = 'inactive'
}

// ✅ After: as const has zero runtime cost
const Status = {
  Active: 'active',
  Inactive: 'inactive'
} as const;
type Status = typeof Status[keyof typeof Status];
```
