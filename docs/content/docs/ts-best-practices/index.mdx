## What It Helps You Do

Use this skill to improve TypeScript and JavaScript code quality with patterns that reduce bugs, tighten type safety, and make behavior easier to reason about.

It is especially useful when you need to:
- Review TS/JS code for correctness, safety, and maintainability issues
- Replace weak patterns like `any`, `enum`, nullable returns, or mutation-heavy state updates
- Refactor control flow, function design, naming, and error handling into clearer, safer defaults
- Audit files or directories for recurring TypeScript anti-patterns before they spread

## When to Use

Use this skill when:
- Writing or reviewing `.ts` or `.js` code
- Fixing type errors or trying to avoid weak typing in new code
- Refactoring state management, return values, validation, or control flow
- Performing a code-quality review focused on safety and correctness rather than speed
- You want a structured TypeScript audit or direct guidance on specific TS/JS patterns

## How It Works

- Focuses on defensive programming patterns such as explicit validation, safer return shapes, bounded logic, and clearer error paths.
- Encourages predictable TypeScript usage, including stronger typing choices and patterns that avoid fragile or misleading abstractions.
- Uses concrete wrong-vs-right examples so the guidance is practical during both code reviews and direct implementation.
- Adjusts to the scope of the request: larger reviews can become structured audits, while narrow requests stay focused on the code in front of you.

## Examples

### Example: Auditing a file for TypeScript anti-patterns

```bash
/accelint-ts-best-practices src/utils/parse-config.ts
```

You get a review of correctness and type-safety issues, with recommended replacements for risky patterns.

### Example: Reviewing a directory before a refactor

```bash
/accelint-ts-best-practices src/lib
```

You get a broader audit that helps prioritize cleanup work across multiple files.

## Good to Know

> **Good to know:** This skill is about correctness, safety, and maintainability. For runtime speed, allocation issues, or V8-specific tuning, use `accelint-ts-performance` instead.

> **Good to know:** For JSDoc, code comments, and documentation-focused cleanup, use `accelint-ts-documentation` instead.

> **Good to know:** Broad audit requests can produce a severity-based report, while narrow fix requests usually get direct implementation guidance instead of a formal report.
