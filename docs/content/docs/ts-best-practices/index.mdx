## Usage

Activate this skill by:
- Using the `/accelint-ts-best-practices` command with optional path
- Saying "review this TypeScript code", "audit code quality", "check for anti-patterns"

```bash
# Review a single file
/accelint-ts-best-practices src/utils/parser.ts

# Audit an entire directory
/accelint-ts-best-practices src/api/

# Review code in current context
/accelint-ts-best-practices
```

Once activated, the skill analyzes your code for type safety issues, defensive programming gaps, and correctness problems. You get side-by-side before/after examples for each finding.

## When to Use

Use this skill when:
- Writing or reviewing TypeScript/JavaScript code
- Fixing type errors or avoiding `any`/`enum`/`null`
- Implementing control flow, state management, or error handling
- Conducting code reviews for anti-patterns

Don't use for:
- **Performance optimization** → use `accelint-ts-performance` instead
- **Documentation** → use `accelint-ts-documentation` instead

## What It Checks

**Type Safety** - `any` types, `enum` usage, type vs interface, bundler paths

**Safety** - Input validation, unbounded loops, error handling, assertions

**Code Quality** - Naming conventions, function size, return values, immutability, pure functions, parameter mutation

## Examples

### Example: File Audit

```bash
# Command
/accelint-ts-best-practices src/api/handler.ts

# Output: Structured report

Executive Summary:
Reviewed handler.ts. Found 3 issues:
- 1 Critical (any type)
- 1 High (null return)
- 1 Medium (parameter mutation)

Critical Issues:

1. Avoid `any` Type
   Line: 15
   Severity: Critical
   
   ❌ Before:
   function handle(req: any) {
     return req.body.data;
   }
   
   ✅ Fix:
   function handle(req: Request) {
     return req.body.data;
   }

2. Return Zero Values
   Line: 42
   Severity: High
   
   ❌ Before:
   function findUser(id: string): User | null {
     return users.find(u => u.id === id) ?? null;
   }
   
   ✅ Fix:
   function findUser(id: string): User {
     return users.find(u => u.id === id) ?? { id: '', name: '', email: '' };
   }
```

### Example: Inline Review

```bash
# Command
/accelint-ts-best-practices

# Then paste code:
enum Status {
  Active = 'active',
  Inactive = 'inactive'
}

# Output: Direct fix

Replace `enum` with `as const` object:

✅ Fix:
const Status = {
  Active: 'active',
  Inactive: 'inactive'
} as const;

type Status = typeof Status[keyof typeof Status];

Why: `enum` generates extra JavaScript at runtime. `as const` provides
the same type safety with zero overhead.
```

## Common Pitfalls

> **Good to know:** This skill focuses on code correctness and type safety, not performance. For optimization tasks (reducing allocations, improving loop efficiency), use `accelint-ts-performance` instead.

> **Good to know:** When you scan a file or directory with the slash command, you get a formal audit report with severity levels. For inline questions about specific code, the skill gives direct fixes without the report structure.

> **Good to know:** The skill delegates documentation tasks to `accelint-ts-documentation`. If you're adding JSDoc comments or fixing comment quality, use that skill instead.

> **Good to know:** For systematic audits across multiple related skills (testing, performance, documentation), use `accelint-ts-audit-all` which runs all TypeScript audits in sequence.

## Report Format

When scanning files or directories with the slash command, the skill generates:

**Executive Summary** - Issue counts by severity (Critical, High, Medium, Low)

**Categorized Findings** - Grouped by Type Safety, Safety, State Management, Code Quality

**Before/After Examples** - Code snippets showing the problem and solution with line numbers

**Impact Analysis** - Potential bugs, type safety gaps, maintainability issues

**Phase 2 Summary** - Tracking table for all issues across audit iterations

For inline questions (without slash command), the skill skips the formal report and provides direct fixes with brief explanations.
