## What It Helps You Do

Use this skill to implement TanStack Query correctly in React applications with Next.js App Router and Server Components. It helps you:

- Configure QueryClient safely for server-rendered applications without data leakage
- Structure query keys for consistent cache invalidation
- Choose between optimistic and pessimistic mutation patterns
- Debug performance issues with high observer counts
- Integrate server-side caching with client-side state management
- Handle hydration correctly with HydrationBoundary

The skill is especially useful when you're setting up a new QueryClient, debugging why data isn't updating after saves, dealing with N+1 query problems in lists, or implementing multi-layer caching with Next.js `use cache`.

## When to Use

Use this skill when:

- Configuring QueryClient for Next.js App Router
- Implementing mutations with useMutation
- Debugging performance issues in components using TanStack Query
- Adding optimistic updates
- Working with useSuspenseQuery, useQuery, invalidateQueries
- Integrating HydrationBoundary for server rendering
- Setting up multi-layer caching strategies
- Seeing infinite requests, stale data flashes, or hydration mismatches

### Core Patterns

**Server safety**: Ensures QueryClient setup prevents data leakage between users by using factory functions instead of singletons on the server.

**Query key architecture**: Hierarchical key factories enable precise cache invalidation at any level—invalidate everything, all lists, or a single item.

**Mutation patterns**: Decision matrix for choosing optimistic vs pessimistic updates based on risk, reversibility, and user experience requirements.

**Observer economics**: Identifies when too many components subscribing to the same query creates performance bottlenecks and provides refactoring patterns.

**Cache configuration**: Decision tables for staleTime, gcTime, and refetchInterval tuned to data freshness requirements—from static lookup tables to real-time dashboards.

## Key Decisions

### Cache Configuration

Match configuration to your data's freshness requirements:

| Data Type | staleTime | gcTime | When |
|-----------|-----------|--------|------|
| Reference data (countries, categories) | 1 hour | Infinity | Changes rarely, high cache retention |
| User profiles | 5 minutes | 10 minutes | Moderate freshness needs |
| Real-time tracking | 5 seconds | 30 seconds | Frequent updates required |
| Search results | 1 minute | 5 minutes | Cacheable but not time-critical |

### Mutation Strategy

Choose your approach based on reversibility and stakes:

| Pattern | When to Use |
|---------|-------------|
| **Optimistic** | Toggle switches, drag-and-drop, low-stakes UI that needs instant feedback |
| **Pessimistic** | Multi-step forms, batch operations, life-critical systems, audit trails |

### Observer Thresholds

Monitor observer counts to catch performance issues:

| Observer Count | Action |
|----------------|--------|
| 1-20 | Normal, no action needed |
| 21-50 | Consider hoisting queries to parent components |
| 51-100 | Refactor required—hoist queries or use select |
| 100+ | Critical—immediate refactor to single query with prop distribution |

## Common Fixes

**Data doesn't update after save**: You likely copied query data into useState. Use the query data directly or derive it with useMemo—TanStack Query handles updates automatically.

**Infinite requests**: Query keys are unstable. Avoid `Date.now()`, unsorted arrays, or non-deterministic object keys. Use factory functions with stable serialization.

**N duplicate requests in lists**: Each list item is calling useQuery individually. Hoist the query to the parent component and pass data down as props.

**Query fires with undefined params**: Add an `enabled` guard: `useQuery({{ queryKey, queryFn, enabled: Boolean(dependency) }})`

**Optimistic update won't rollback**: Your `onError` callback isn't using the context from `onMutate`. Always pair `onMutate` with `onSettled` for cleanup.

**Server hydration mismatch**: Timestamps or user-specific data differ between server and client. Use `suppressHydrationWarning` or ensure deterministic rendering with stable timestamps.

## Multi-Layer Caching

When using Next.js `use cache` alongside TanStack Query, use the same key factory for both layers:

```typescript
// Shared key factory
export const keys = {{
  all: () => ['tracks'] as const,
  detail: (id: string) => [...keys.all(), id] as const,
}};

// Server mutation invalidates server cache
updateTag(keys.detail(id).tag);

// Client mutation invalidates client cache
queryClient.invalidateQueries({{ queryKey: keys.detail(id) }});
```

This keeps server cache (cross-request) and client cache (per-tab) synchronized with the same invalidation hierarchy.

## Critical Patterns

**Never use a singleton QueryClient on the server**: Each request needs its own isolated QueryClient instance to prevent data from one user leaking to another.

**Never synchronize query data to useState**: Background refetches and cache updates won't reflect in your local copy, causing "my save didn't work" bugs.

**Never put queries inside list item components**: Creates N network requests and N observers, causing O(n) performance overhead on cache updates.

**Always use cancelQueries before optimistic updates**: Background refetches can overwrite your optimistic changes before the mutation completes.

> **Good to know:** For static reference data that never changes, consider whether TanStack Query's refetching machinery adds value over simpler alternatives like module-level constants or Context.
