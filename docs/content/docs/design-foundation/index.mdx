## What It Helps You Do

Use this skill to style components with `@accelint/design-foundation` and `@accelint/design-toolkit` using the design system's Tailwind conventions rather than vanilla Tailwind defaults.

It is especially useful when you need to:
- Apply semantic tokens for color, spacing, and outlines
- Structure styles in CSS modules with the expected layer and variant patterns
- Set up the required PostCSS and `@reference` wiring
- Migrate components from vanilla Tailwind to design-foundation conventions

> **Good to know:** This is not vanilla Tailwind. The design foundation replaces default Tailwind theme assumptions with semantic tokens, 1:1 numeric spacing, and outline-first borders.

## When to Use

Use this skill when:
- Styling components with `@accelint/design-foundation` or `@accelint/design-toolkit` packages
- Users say "style this", "add styling", "theme this component", "add colors", "add spacing"
- Working with `.module.css` files or CSS modules
- Seeing "CSS modules", "setup design foundation", "@variant" mentioned
- Encountering build errors like "undefined variable" or "@variant not found"

## Prerequisites

Required setup before using design foundation:

**PostCSS configuration** - Create `postcss.config.mjs` in project root:
```javascript
export default {
  plugins: {
    '@tailwindcss/postcss': {},
    '@accelint/postcss-tailwind-css-modules': {}, // Required
  },
};
```

**CSS module references** - Every `.module.css` file must include:
```css
@reference '@accelint/design-foundation/styles';
```

**Root layout import** - Import design foundation styles first:
```tsx
import '@accelint/design-foundation/styles'; // First import
import { ReactNode } from 'react';

export default function RootLayout({ children }: { children: ReactNode }) {
  return <html>{children}</html>;
}
```

## How It Works

The skill steers you toward three core conventions:

### Semantic Tokens

Prefer semantic tokens over primitive values so styles adapt to theme changes:
- `bg-*` for surfaces and backgrounds
- `fg-*` for text and foreground emphasis
- `outline-*` for borders, dividers, and focus treatments

```tsx
<div className="bg-surface-default fg-primary-bold outline-1 outline-interactive">
  Content
</div>
```

### Semantic Spacing First

Use the semantic spacing scale (`xxs` → `oversized`) for normal component work, and fall back to numeric spacing only when the design truly requires an exact value.

```css
.button {
  @apply px-m py-xs;
}

.nonStandard {
  @apply p-13;
}
```

### Outline-First Borders

Prefer `outline-*` utilities over `border-*` so visual boundaries do not change element dimensions.

```css
.card {
  @apply outline-1 outline-interactive;
}
```

## Component Styling Pattern

Component styles belong in CSS modules, not inline classes:

**button.module.css:**
```css
@reference '@accelint/design-foundation/styles';

@layer components.l1 {
  .button {
    @apply px-m py-xs bg-interactive-bold fg-inverse-bold;
  }
}

@layer components.l2 {
  .button {
    @variant color-critical {
      @apply outline-2 outline-critical-bold fg-critical-bold;
    }

    @variant size-large {
      @apply px-l py-s;
    }
  }
}
```

**button.tsx:**
```tsx
import { clsx } from '@accelint/design-foundation/lib/utils';
import styles from './button.module.css';

export function Button({ children, color, size }) {
  return (
    <button
      className={styles.button}
      data-color={color}
      data-size={size}
    >
      {children}
    </button>
  );
}
```

> **Good to know:** Always import `clsx` from `@accelint/design-foundation/lib/utils`, not directly from the `clsx` package. The design foundation re-exports it with additional type support.

## Good to Know

> **Good to know:** Every `.module.css` file needs `@reference '@accelint/design-foundation/styles';` at the top. Missing it usually causes token and variant errors that look unrelated at first glance.

> **Good to know:** Keep each rule to a single `@apply` statement when possible. That preserves IDE plugin support better than splitting utilities across multiple `@apply` lines.

> **Good to know:** Use `@variant` blocks instead of handwritten attribute selectors for component states and variants. That keeps your styling aligned with the design foundation's conventions.

> **Good to know:** If tokens show up as raw values or selectors like `group-hover/button` fail, the usual cause is setup drift: a missing PostCSS plugin, a missing `@reference`, or importing `@accelint/design-foundation/styles` too late.

> **Good to know:** Component styles belong in CSS modules (`.module.css`), not inline `className` props. Use inline Tailwind classes only for minor one-off overrides.

> **Good to know:** Semantic tokens should be your first choice. The utility classes fall back to `domain-*` and `primitive-*` tokens for rare cases where designs exceed the system.
