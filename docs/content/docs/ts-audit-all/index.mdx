## What It Helps You Do

Use this skill to systematically audit TypeScript files for quality, performance, and documentation issues across multiple dimensions. It orchestrates four specialized audit skills in a controlled sequence with interactive approval at each stage.

This skill is especially useful when you need to:
- Audit entire directories of TypeScript code systematically
- Apply multiple audit perspectives (testing, best practices, performance, documentation) without contradiction
- Track audit progress across multiple sessions when working on large codebases
- Review and approve changes interactively before they are applied
- Work on audits in parallel using isolated git worktrees

Activate this skill by using the `/accelint-ts-audit-all` command followed by a file or directory path.

## When to Use

This is a command-only skill with no natural language triggers. Invoke it explicitly when you want to:
- Run a comprehensive audit on TypeScript files
- Apply multiple code quality checks systematically
- Track audit progress across multiple files
- Work through audit findings with interactive approval

## How It Works

The skill executes an 8-step audit process for each file:

**Phase 1: Test Coverage First**
- Runs testing audit to establish test coverage before making any code changes
- Includes 100-pass verification for property-based tests to ensure stability
- Interactive approval with severity-ranked issue table

**Phase 2: Code Quality &amp; Performance in Parallel**
- Runs best-practices and performance audits simultaneously to identify overlapping or conflicting recommendations
- Presents merged suggestions when changes affect the same code
- Interactive approval with combined severity table from both audits

**Phase 3: Verification**
- Runs documented test, build, and lint commands
- Ensures changes don't break existing functionality
- Interactive approval for any verification failures

**Phase 4: Documentation**
- Runs documentation audit after code changes are stable
- Adds JSDoc comments, TODO markers, and improves code comments
- Final verification to ensure documentation didn't introduce issues

The skill creates an isolated git worktree for audit work to enable parallel audits without conflicts. All progress is tracked in audit process files that survive context limits.

### Interactive Approval Pattern

Every change goes through a two-phase approval process:

1. **Overview Table** - Displays all issues with emoji severity indicators (🛑 Critical, ⚠️ High, ⚡ Medium, 🔵 Low, ✅ Info)
2. **Detailed Changes** - Shows complete before/after code for each issue with expected benefits
3. **Numbered Selection** - User specifies which changes to apply (e.g., "1, 2, 5" or "all" or "skip")

This pattern prevents one-by-one approvals and allows you to spot conflicts across parallel processes.

### Progress Tracking

The skill maintains detailed progress files:
- **Audit Process File** - Tracks current file, completed steps, verification commands
- **Audit History File** - Archives detailed findings after each file completes
- **Worktree Management** - Isolates audit work in timestamped branches

Progress is saved after each step, allowing you to pause and resume audits across sessions without losing context.

## Prerequisites

- Git repository with a valid branch
- TypeScript files to audit (.ts files, excluding .test.ts, .spec.ts, .bench.ts)
- Documented verification commands (test, build, lint) in your project
- Clean working directory before starting (commit or stash changes)

## Good to Know

&gt; **Good to know:** This skill creates isolated git worktrees to prevent conflicts. Multiple audits can run in parallel without interfering with each other or your main branch.

&gt; **Good to know:** The skill maintains `.agents/audit/` tracking files in the original repository (should be gitignored) while performing work in `.agents/worktrees/` directories.

&gt; **Good to know:** Property-based tests require 100 consecutive passes to verify stability. The skill runs verification without coverage reporting to increase speed and prevent flaky failures.

&gt; **Good to know:** Best-practices and performance audits run in parallel to identify contradictory recommendations before presenting them. The skill will merge overlapping suggestions when possible.

## What You Get

After completion:
- Comprehensive audit of all TypeScript files in the target directory
- Interactive approval history showing which changes were accepted or rejected
- Detailed audit history file with findings from each file
- Merged branch with all approved changes committed to your original branch
- Clean worktree removal after successful merge

## Examples

### Example: Auditing a Single File

```bash
/accelint-ts-audit-all src/utils/parser.ts
```

The skill will:
1. Create an isolated worktree with timestamped branch
2. Run the 8-step audit process
3. Present findings with severity indicators
4. Apply approved changes interactively
5. Merge back to your original branch

### Example: Auditing an Entire Directory

```bash
/accelint-ts-audit-all src/components
```

The skill will:
1. Discover all .ts files (excluding test files)
2. Create a TODO list in the audit process file
3. Audit each file systematically with progress tracking
4. Save progress after each file to survive context limits
5. Allow resumption across multiple sessions

### Example: Resuming an Existing Audit

If you have an existing audit in progress:

```bash
/accelint-ts-audit-all
```

The skill will:
1. Check `.agents/audit/` for existing audit process files
2. Display current status and resume instructions
3. Continue from the last saved checkpoint
4. Work in the existing worktree without creating a new one

## Limits

- This skill orchestrates other audit skills but does not implement new audit logic itself
- It does not run audits on test files (.test.ts, .spec.ts) or benchmark files (.bench.ts)
- It requires interactive approval and cannot run fully autonomously
- Large directories (&gt;5 files) typically require multiple sessions due to context limits
