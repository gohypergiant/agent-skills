## What It Helps You Do

Use this skill to plan features and changes through the QRSPI methodology before writing any code. It automates the Question generation → Research → Design → Structure phases, producing a complete OpenSpec change ready for implementation.

This skill is especially useful when you need to:
- Break down a ticket or feature request into structured, implementable tasks
- Ensure research is objective and design decisions are explicit before coding begins
- Create vertical task slices that deliver end-to-end testable increments
- Generate specs-of-record that document both current state and planned changes

Activate this skill by:
- Using the `/accelint-qrspi-propose` command with your ticket or feature description
- Saying "plan this with QRSPI", "use QRSPI to plan", "start QRSPI workflow"
- Asking to "create spec-driven change" or "plan a feature before implementation"

> **Good to know:** This skill handles ONLY the planning phase. After completion, you'll run `/accelint-qrspi-apply` to begin implementation. This separation allows you to plan multiple changes before implementing any of them.

## When to Use

Use this skill when:
- You want to plan a ticket, feature, or change before writing code
- You need objective research about the current codebase state
- You want explicit design decisions documented before implementation
- You're working in a spec-driven development workflow with OpenSpec
- You need task breakdown with vertical slicing for parallel implementation

## Prerequisites

- OpenSpec installed and initialized with an `openspec/` directory
- Expanded OpenSpec workflows enabled (`explore`, `new`, `continue`)
- Project configuration in `openspec/config.yaml`
- Agent behavior context in `AGENTS.md` or `CLAUDE.md`

If the expanded workflows are missing, enable the expanded OpenSpec profile before using this skill.

## How It Works

The skill moves through the QRSPI phases with two required review checkpoints:

1. **Questions** — turns the request into research questions without jumping to solutions
2. **Research** — gathers objective facts about the current codebase
3. **Design** — creates `proposal.md` and `design.md` from the research
4. **Design checkpoint** — you review the proposed direction before specs are written
5. **Specs &amp; Tasks** — generates capability deltas and a task plan
6. **Tasks checkpoint** — you approve the task structure and parallelization plan before implementation

The task plan is intentionally vertically sliced, so each slice delivers an end-to-end increment rather than a layer-by-layer handoff.

## What You Get

A complete OpenSpec change under `openspec/changes/<change-name>/`, including:
- `proposal.md`
- `design.md`
- capability delta specs
- `tasks.md` with vertical slices and a parallelization plan

The result is ready for implementation via `/accelint-qrspi-apply <change-name>`.

## Examples

### Example: Planning a CLI feature

```bash
/accelint-qrspi-propose

ATI-12: smart-ls CLI tool

Create a CLI tool that returns structured directory listings as JSON.
Should support filtering by file type and sorting by size or date.
```

The skill turns the request into research questions, produces a design for review, then generates specs and vertically sliced tasks.

### Example: Turning a vague request into a planned change

```bash
/accelint-qrspi-propose

Users complain that search is slow. We should make it faster.
```

The skill researches the current search implementation, proposes a direction for review, and turns the approved design into a task plan.

## Good to Know

> **Good to know:** The two review checkpoints are central to the workflow. They let you correct direction before implementation starts.

> **Good to know:** This skill works best when `openspec/config.yaml` already defines clear design, spec, and task rules.

> **Good to know:** Task subtasks use markdown checklist format (`- [ ] instruction`) so the apply workflow can track progress later.

## Limits

This skill does not implement code, run tests, create pull requests, or archive completed changes.

For implementation, use `/accelint-qrspi-apply <change-name>` after planning is complete. For archiving, use `/accelint-qrspi-archive <change-name>` after implementation and review.
