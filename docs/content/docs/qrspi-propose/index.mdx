## Usage

Activate this skill by:
- Using the `/accelint-qrspi-propose` command with a ticket or feature description
- Saying "plan this with QRSPI", "use QRSPI to plan", "start QRSPI workflow", or "create spec-driven change"
- Asking to plan a feature/change before implementation

```bash
# With a ticket description
/accelint-qrspi-propose ATI-123: Add user authentication to the API...

# Interactive (will prompt for ticket)
/accelint-qrspi-propose
```

Once activated, the skill orchestrates the QRSPI methodology:
1. **Questions** - Generates research questions from the ticket
2. **Research** - Answers questions by examining codebase and existing specs
3. **Design** - Creates proposal.md and design.md artifacts
4. **Checkpoint 1** - MANDATORY review of design.md before continuing
5. **Structure** - Generates delta specs and tasks.md with vertical slicing
6. **Checkpoint 2** - MANDATORY review of tasks.md before completion

Output is a complete OpenSpec change ready for `/accelint-qrspi-apply`.

## When to Use

Use this skill when:
- Planning a ticket or feature request before implementation
- Starting spec-driven development with QRSPI methodology
- Creating an OpenSpec change with structured research and design
- Working on complex features that benefit from upfront planning

This skill handles ONLY the planning phase—it does NOT implement code. After completion, continue with `/accelint-qrspi-apply` for implementation.

## Prerequisites

Requires OpenSpec's expanded workflows (`explore`, `new`, `continue`).

Verify configuration:

```bash
openspec config list
```

If any workflows are missing, enable the expanded profile:

```bash
openspec config profile
# Select "expanded" from the list
openspec update
```

The skill validates prerequisites automatically and guides you if setup is needed.

## Key Patterns

### Context Isolation

QRSPI prevents "solution-first thinking" through context isolation:

- **Questions stage**: Ticket IN context → generates research questions
- **Research stage**: Ticket OUT of context, only questions IN context → objective facts
- **Design stage**: Questions + research IN context, ticket OUT → prevents completion bleed

This ensures research is objective and design is grounded in facts, not assumptions.

### Mandatory Checkpoints

Two critical review gates before code is written:

**Checkpoint 1 (After design.md)**:
- Review pattern references (did it find the current way or the legacy way?)
- Check assumptions (are there unresolved decisions presented as facts?)
- Verify affected systems (are all service boundaries accounted for?)
- Catch scope creep (does the design exceed the ticket?)

**Checkpoint 2 (After tasks.md)**:
- Validate vertical slicing (end-to-end features, not architectural layers)
- Review parallelization strategy (can slices run independently?)
- Verify testability (does each slice have explicit verification?)

Corrections at these checkpoints cost minutes; the same corrections after implementation cost review cycles.

### Vertical Slicing

The skill enforces vertical slicing for parallelization and incremental delivery. Each slice must deliver an end-to-end testable feature path.

✓ **Correct - Vertical (end-to-end feature slices)**:
```
Slice 1: Mock API + working frontend (user can see and click)
Slice 2: Wire real service layer (now pulls actual data)
Slice 3: Add database integration (data persists)
```

✗ **Wrong - Horizontal (architectural layers)**:
```
Slice 1: All database migrations
Slice 2: All service layer changes
Slice 3: All API endpoints
Slice 4: All frontend components
```

If the OpenSpec workflow generates horizontal slices, the skill automatically converts them to vertical slices before presenting for review.

### Frontmatter Capture

After design approval, the skill captures `specs_touched` and `decisions` as structured frontmatter in design.md. This metadata enables the `accelint-qrspi-archive` skill to cross-link capabilities and maintain running indices.

### Example: Converting Horizontal Slices

If the OpenSpec workflow generates horizontal slicing (organized by architectural layer), the skill automatically restructures to vertical slicing:

**Before (horizontal)**:
```
Slice 1: Database Schema Changes
- [ ] Add theme column to users table
- [ ] Create theme_preferences table

Slice 2: API Layer Changes
- [ ] Add GET /api/user/theme endpoint
- [ ] Add POST /api/user/theme endpoint

Slice 3: Frontend Changes
- [ ] Create ThemeSwitcher component
- [ ] Wire theme context provider
```

**After (vertical)**:
```
Slice 1: Theme Toggle with Mock Data
Deliverable: Working theme switcher UI with localStorage persistence
- [ ] Create ThemeSwitcher component
- [ ] Add theme context provider
- [ ] Implement localStorage persistence
Test: User can toggle dark/light mode, preference survives page refresh

Slice 2: Theme API Integration
Deliverable: Theme preference synced to backend
- [ ] Add GET /api/user/theme endpoint
- [ ] Add POST /api/user/theme endpoint
- [ ] Wire ThemeSwitcher to API
Test: Theme preference syncs across devices after login

Slice 3: Database Persistence
Deliverable: Theme preference persists server-side
- [ ] Add theme column to users table
- [ ] Update user service to read/write theme
Test: Theme preference survives account migration
```

## Common Pitfalls

> **Good to know:** This skill stops after planning—it does NOT implement code. After completion, run `/clear` to start fresh context, then use `/accelint-qrspi-apply` for implementation. Keeping planning and implementation in separate contexts prevents token bloat and maintains focus.

> **Good to know:** The mandatory checkpoints (after design.md and after tasks.md) are not optional. If you skip review or proceed without explicit approval, you bypass QRSPI's core value—cheap corrections before code is written.

> **Good to know:** Context isolation is critical. The ticket should NOT be visible during research or design phases. If you see the ticket in those contexts, the agent may propose solutions instead of gathering objective facts.

> **Good to know:** Vertical slicing is required for the `/accelint-qrspi-apply` skill to parallelize implementation. Each slice must deliver an end-to-end testable feature, not an architectural layer. The skill enforces this and converts horizontal slices automatically.

> **Good to know:** The skill captures `specs_touched` and `decisions` frontmatter ONLY after design.md is approved. This metadata enables `accelint-qrspi-archive` to cross-link capabilities. If design.md lacks clear capability declarations or decision rationale, the skill will ask you to add it before continuing.
