## What It Helps You Do

Use this skill to audit and optimize JavaScript/TypeScript code for performance. It systematically identifies bottlenecks using V8 profiling patterns and applies proven optimization techniques.

The skill is especially useful when you need to:
- Find and fix performance anti-patterns (O(n²) complexity, excessive allocations, I/O blocking)
- Reduce runtime for slow functions, rendering pipelines, or real-time systems
- Audit code for performance issues regardless of current usage context
- Apply V8-specific optimizations for monomorphic/polymorphic code
- Balance performance gains against code complexity and maintainability

Activate this skill by:
- Using the `/accelint-ts-performance` command with a file or directory path
- Saying "optimize performance", "audit performance", "this is slow", "reduce allocations"
- Working on code that needs to run faster or handle larger datasets

## When to Use

Use this skill when:
- Users say "optimize performance", "audit performance", "this is slow", "reduce allocations", "improve speed", "check performance"
- Analyzing code for performance anti-patterns like O(n²) complexity, excessive allocations, or I/O blocking
- Optimizing functions that may be called in hot paths—utilities, formatters, parsers are often called in loops or rendering pipelines
- Fixing V8 deoptimization issues (monomorphic/polymorphic problems, inline caching failures)
- You need systematic performance analysis with expected gain estimates

> **Good to know:** This skill focuses exclusively on runtime performance optimization. For general TypeScript best practices like type safety, avoiding `any`/`enum`, or defensive programming patterns, use the `accelint-ts-best-practices` skill instead.

## How It Works

The skill uses a 4-phase workflow that combines static code analysis with profiling-guided optimization:

### Two Operating Modes

**Audit Mode** (when explicitly invoked):
- Generates structured audit reports using a standard template
- Reports all findings for user review before implementation
- User decides which optimizations to apply based on their context

**Implementation Mode** (during feature work):
- Identifies and applies optimizations directly inline
- No formal report needed
- Focuses on fixing issues as they're discovered

### 4-Phase Optimization Process

**Phase 1: Profile to Identify Bottlenecks**

Audits ALL code for performance anti-patterns through systematic static analysis. Does not skip code based on assumptions about usage frequency—utilities, formatters, and parsers are often called in hot paths.

When profiling tools are available, uses them to establish baselines:
- Browser: Chrome DevTools Performance tab
- Node.js: `node --prof` with flame graphs

Identifies issues including:
- O(n²) complexity from nested loops or repeated searches
- Excessive allocations from template literals, object spreads, array method chains
- Array method chaining (`.filter().map().reduce()`)
- Blocking async operations
- Try/catch in hot loops

**Phase 2: Analyze and Categorize Issues**

Categorizes every identified issue by optimization type with expected performance gains:

| Issue Type | Category | Expected Gain |
|------------|----------|---------------|
| Nested loops, O(n²) complexity | Algorithmic optimization | 10-1000x |
| Repeated expensive computations | Caching &amp; memoization | 2-100x |
| Allocation-heavy code | Allocation reduction | 1.5-5x |
| Sequential access violations | Memory locality | 1.5-3x |
| Excessive I/O operations | I/O optimization | 5-50x |
| Blocking async operations | I/O optimization | 2-10x |

**Phase 3: Optimize Using Performance Patterns**

Applies proven optimization patterns from detailed reference files organized by category:
- **Algorithmic**: Reduce looping (use Map/Set for O(1) lookups), reduce branching
- **Caching**: Memoization, cache property access in loops
- **I/O**: Batching, defer await until actually needed
- **Memory**: Avoid allocations, optimize object operations
- **Locality**: Predictable execution patterns for CPU cache efficiency
- **Safety**: Bounded iteration to prevent runaway loops

Each pattern includes ❌/✅ examples showing before/after code.

**Phase 4: Verify Improvements**

Measures actual performance gains and verifies correctness:
- Re-runs profiler with same inputs
- Documents speedup factor (e.g., "2.3x faster")
- Runs full test suite to catch optimization bugs
- Adds tests for edge cases affected by changes

Keeps optimizations based on speedup and maintainability:
- &gt;10x speedup: Always keep if tests pass
- 2-10x speedup: Keep if tests pass and code remains maintainable
- 1.2-2x speedup: Keep for hot paths or real-time systems (60fps rendering)
- &lt;1.05x speedup: Revert unless trivial or critical hot path

### Audit Everything Philosophy

The skill audits ALL code for performance anti-patterns, regardless of how simple a function appears. Utility functions, formatters, parsers, and validators are frequently called in loops, rendering pipelines, or real-time systems even when their implementation seems straightforward.

Whether profiling data is available or not, the skill performs systematic static analysis to identify and report all anti-patterns with expected gains. It doesn't filter based on perceived "severity"—it reports everything found so users can make informed decisions about which optimizations to prioritize.

### Adaptive Guidance

The skill calibrates its prescriptiveness based on optimization impact:

| Optimization Type | Guidance Approach | Example |
|------------------|------------------|---------|
| **Algorithmic (10x+ gain)** | Multiple valid approaches based on constraints | "Use Map for O(1) lookup or Set for deduplication" |
| **Caching (2-10x gain)** | Pattern with examples and cache invalidation strategy | "Memoize with WeakMap if lifecycle matches source objects" |
| **Micro-optimization (1.1-2x)** | Exact pattern, measure first | "Cache array.length in loop: `for (let i = 0, len = arr.length; ...)`" |

## Examples

### Example: Auditing a File for Performance Issues

```bash
/accelint-ts-performance src/utils/data-processor.ts
```

The skill generates a structured audit report identifying:
- O(n²) nested loop in data filtering (expected gain: 100x)
- Array method chaining creating intermediate arrays (expected gain: 3x)
- Property access in loop that should be cached (expected gain: 1.5x)

Each issue includes the specific pattern, expected performance gain, and which reference file contains the fix.

### Example: Optimizing During Feature Development

While implementing a new feature, the skill notices:

```typescript
// ❌ Before: O(n²) - nested iteration
for (const user of users) {
  const items = allItems.filter(item => item.userId === user.id);
  process(items);
}
```

Applies optimization directly:

```typescript
// ✅ After: O(n) - single pass with Map lookup
// Performance: reduce-looping.md - build lookup once pattern
const itemsByUser = new Map<string, Item[]>();
for (const item of allItems) {
  if (!itemsByUser.has(item.userId)) {
    itemsByUser.set(item.userId, []);
  }
  itemsByUser.get(item.userId)!.push(item);
}

for (const user of users) {
  const items = itemsByUser.get(user.id) ?? [];
  process(items);
}
```

### Example: Real-Time System Optimization

For 60fps rendering (16.67ms frame budget):

```bash
/accelint-ts-performance src/rendering/animation-loop.ts
```

Identifies even 1.05x improvements in critical hot paths:
- Caching array.length in animation loop
- Removing try/catch that prevents V8 inlining
- Sequential array access for CPU cache locality

Documents frame timing impact for each optimization.

## Good to Know

> **Good to know:** This skill handles runtime performance optimization. For general code quality issues like type safety (`any`, `enum`, `null`), naming conventions, or defensive programming patterns, use `accelint-ts-best-practices` instead.

> **Good to know:** The skill audits code systematically regardless of whether profiling data is available. When profilers aren't accessible, it performs static analysis to identify all anti-patterns and reports them with expected gains.

> **Good to know:** Performance optimizations can introduce subtle bugs in edge cases. The skill emphasizes adding comprehensive tests before optimizing and verifying correctness after changes. If tests fail, the optimization is reverted—performance bugs are still bugs.

> **Good to know:** For real-time systems (60fps rendering, live data visualization), even 1.05x improvements in critical hot paths can be valuable. The skill uses frame timing profilers to verify impact on frame budgets rather than just measuring total execution time.

## Prerequisites

- Git repository for SHA tracking
- Optional: Profiling tools for baseline measurements
  - Browser: Chrome DevTools Performance tab
  - Node.js: `node --prof` and flame graph generators
- Test suite for verifying correctness after optimizations

## Limits

This skill focuses exclusively on runtime performance optimization:
- **Does not handle** general TypeScript/JavaScript best practices (type safety, naming conventions, defensive programming)
- **Does not optimize** code you don't own (shared utilities, library internals, code under active development by others)
- **Does not profile automatically** in all environments—you may need to run profilers separately and provide data

For optimizations spanning multiple files or coordinating with team members, the skill identifies issues but leaves coordination to you.
