## Usage

Activate this skill by:
- Using the `/accelint-ac-to-playwright` command
- Saying "review these AC", "evaluate if AC are ready for automation", "convert these AC to tests", "generate Playwright tests from acceptance criteria"
- Mentioning "turn AC into tests", "check if AC can be converted", "assess AC quality"
- Working with `.md` bullet-style or `.feature` Gherkin acceptance criteria files

The skill operates in two modes:

**Assessment mode** - Analyzes AC files for conversion readiness without generating artifacts:
```bash
# Activate with phrases like:
# "review these AC"
# "are these AC automation-ready?"
# "check if these AC can be converted"
```

**Full conversion mode** - Generates JSON test plans and Playwright test files:
```bash
# Activate with phrases like:
# "convert these AC to tests"
# "generate Playwright tests from these AC"
# "turn AC into test automation"
```

Once activated in assessment mode, the skill validates AC structure, targets, actions, and expected outcomes, then reports whether AC are conversion-ready with detailed issue lists.

Once activated in full conversion mode, the skill runs assessment first, then generates validated JSON test plans and translates them to Playwright `.spec.ts` files with shared fixtures.

## When to Use

Use this skill when:
- Reviewing or evaluating acceptance criteria for test automation readiness
- Assessing if AC can be converted as-is or need modifications
- Validating AC quality against Playwright requirements
- Converting `.md` bullet files or `.feature` Gherkin files to Playwright specs
- Generating test automation from requirements
- Creating test plans with structured JSON schema validation

## What It Converts

**Input formats**:
- Bullet-style markdown (`.md`) - each `- ` bullet becomes one test
- Gherkin syntax (`.feature`) - each Scenario becomes one test, Scenario Outlines expand to multiple tests

**Output artifacts**:
- JSON test plans (`<suite-slug>.json`) conforming to schema
- Playwright test files (`<suite-slug>.spec.ts`)
- Shared fixtures (`fixtures/error-handling.ts`, `fixtures/console-tracking.ts`)
- Batch summary JSON with metadata

**Validation checks**:
- Structure and format (proper bullets or Gherkin keywords, step ordering)
- Targets (area/component/intent pattern with controlled vocabulary)
- Actions (recognized verbs mappable to Playwright: click, fill, select, drag)
- Expected outcomes (explicit, measurable assertions)
- Values (quoted literals for fills/selects, not placeholders)

## Key Patterns

### Assessment Workflow

1. Detect intent from user phrasing (review/evaluate/assess)
2. Load `references/acceptance-criteria.md` and `references/test-hooks.md`
3. Analyze each AC file against conversion requirements
4. Report pass/fail with detailed issue list or validated checklist
5. Do not generate any artifacts in this mode

### Conversion Workflow

1. Detect intent from user phrasing (convert/generate/turn into tests)
2. Run assessment mode against all input files first
3. If assessment fails for any file, stop immediately
4. For each passing file:
   - Generate JSON test plan conforming to `references/plan-schema.ts`
   - Validate plan with `npx validate-plan`
   - If validation fails, retry once to fix specific issue, then stop
   - If validation passes, translate plan to Playwright test with `scripts/translate-plan-to-tests.ts`
   - Append summary entry to batch JSON file
5. Copy shared fixtures directory to test output location

### Naming Transformations

| Input | Suite Name | Test Name | Output Slug |
|-------|------------|-----------|-------------|
| `.feature` | `Feature:` text ã lowercase ã capitalize first | Scenario text (lowercase, ~64 char limit) + ` (params)` for Scenario Outlines | suite name ã lowercase, spaces to dashes |
| `.md` | filename ã lowercase ã dashes to spaces ã capitalize first | Summarize bullet intent (present tense, lowercase, ~64 char) | suite name ã lowercase, spaces to dashes |

For Scenario Outlines, parameters use shortest left-to-right column combo that uniquely identifies each row, joined with `/`.

### Target Pattern

Targets must follow the `area.component.intent` pattern with controlled vocabulary:

```
<intent> <component> on the <area>
```

**Example**: "clicks the Submit button on the form" ã `button.form.submit`

**Controlled vocabulary**:
- Area keywords: nav, header, footer, form, drawer, card, toast, modal, table, page, area
- Component keywords: button, link, input, dropdown, checkbox, radio, text, div, component

### Mouse and Keyboard Actions

**Element clicks** (most common):
```
Given the user clicks the Submit button on the form
```

**Coordinate clicks**:
```
Given the user clicks at position 150, 200
Given the user right-clicks at position 300, 400
```

**Keyboard combinations**:
```
Given the user presses Shift+g
Given the user presses Control+Enter
```

Modifier combinations automatically translate to three-step sequence: keyDown ã press ã keyUp.

## Anti-Patterns

**NEVER use bare string values with selectOption** ã Playwright's `selectOption()` matches HTML `value` attributes by default, not visible text. AC writers specify visible option text (e.g., "Premium Plan"), so always use `{{ label: "text" }}` syntax. Using bare strings causes silent mismatches where tests pass locally but fail in production.

**NEVER generate artifacts in assessment mode** ã when the user asks to review/evaluate/assess AC, analyze the AC text only and provide the formatted report. Do not generate JSON plans or test files.

**NEVER skip controlled vocabulary checks** ã verify that area and component keywords in targets match the lists in `test-hooks.md`. Generic targets like `button.generic` bypass the system and create tests that match multiple elements unpredictably.

**NEVER use `goto` action in steps** ã tests start at `startUrl`, navigation happens via clicks or fills that trigger page changes. Using goto mid-test breaks Playwright's navigation lifecycle and causes race conditions.

**NEVER use `doubleClick` for element interactions** ã `doubleClick` is only for coordinate-based double-clicks (x,y positions). For double-clicking elements, use the element-based `click` action twice in sequence.

**NEVER use `mouseClick` for element interactions** ã `mouseClick` is only for coordinate-based clicks (x,y positions). For clicking elements, always use `click` with test IDs.

**NEVER invent assertions** ã only add `expectText`, `expectVisible`, `expectNotVisible` when AC explicitly states expected outcomes (exception: `expectUrl` for navigation, visibility pairs for show/hide actions).

**NEVER assume targets or values** ã if AC says "click the button" without identifying which button, ask for clarification rather than guessing.

**NEVER write a plan file without validating first** ã validation catches structural errors; writing invalid plans creates broken artifacts requiring manual cleanup.

**NEVER process multiple steps of one file in parallel** ã complete the full pipeline (AC ã plan ã test ã summary) for each file before moving to the next to avoid partial artifacts and state confusion.

## Common Pitfalls

> **Good to know:** Assessment mode vs conversion mode intent detection is critical. Phrases like "review", "evaluate", "check if ready", or "assess" trigger assessment only. Phrases like "convert", "generate", or "turn into tests" trigger full conversion. If the user says "review and then convert", run assessment first, report results, then ask if they want to proceed with conversion.

> **Good to know:** When converting AC to tests, the skill always runs assessment mode first. If assessment fails for any file, conversion stops immediately. Fix all assessment issues before attempting conversion.

> **Good to know:** Generated tests import from `fixtures/` directory. When copying tests to your Playwright project, copy both the `.spec.ts` files AND the `fixtures/` directory. Tests will fail to compile without these shared utilities.

> **Good to know:** The skill validates JSON plans before writing files and allows up to 2 attempts total (initial + 1 correction). After 2 failed attempts, it reports the error to the user rather than continuing to retry.

> **Good to know:** Start URL defaults to `/` unless AC explicitly states a different starting page. If tests need to start on a specific page, include it in the AC: "Given the user is on the settings page".
