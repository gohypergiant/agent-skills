## What It Helps You Do

Use this skill to convert acceptance criteria into Playwright test automation. It transforms requirements written in markdown bullets or Gherkin syntax into validated JSON test plans and executable test suites.

It is especially useful when you need to:
- Validate whether AC are ready for automation before writing tests
- Convert AC files (`.md` or `.feature`) into Playwright test suites
- Ensure acceptance criteria follow automation-ready patterns
- Generate test plans with proper target selectors and assertions

Activate this skill by:
- Saying "review these AC", "evaluate AC quality", "check if AC are ready"
- Saying "convert these AC", "generate tests from AC", "turn AC into Playwright tests"
- Mentioning "acceptance criteria" or "Gherkin" with "Playwright" or "automation"

## When to Use

Use this skill when:
- Users ask to review/evaluate/check if AC are ready for automation
- Users want to assess if AC can be converted as-is
- Users ask to validate AC quality for Playwright
- Users want to turn AC into tests, generate tests from acceptance criteria
- Users need to convert `.md` bullets or `.feature` Gherkin files to Playwright specs
- Users request test automation creation from requirements

## How It Works

The skill operates in two modes based on your request:

**Assessment mode** analyzes AC text without generating artifacts. It validates:
- Structure and format (bullet markers, Gherkin syntax, step ordering)
- Target completeness (area/component/intent pattern with controlled vocabulary)
- Action verbs (click, fill, select with proper values)
- Expected outcomes (explicit, measurable assertions)

Reports either "✓ AC are conversion-ready" with validation checklist or "❌ AC are not conversion-ready" with detailed issue list.

**Conversion mode** generates complete test automation:
1. Runs assessment validation first (stops if issues found)
2. Generates JSON test plan from AC using controlled vocabulary
3. Validates plan against schema (max 2 attempts)
4. Translates plan to Playwright spec file
5. Creates summary metadata and copies shared fixtures

### Target Pattern System

The skill expects each action target to follow `area.component.intent`, which keeps selectors predictable and easier to maintain.
- **Area** identifies where the interaction happens
- **Component** identifies the UI element type
- **Intent** identifies the action's purpose

### Output Shape

One AC file becomes one test suite:
- `.md` bullets become individual tests
- `.feature` scenarios become individual tests, while Scenario Outline rows become parameterized cases

## Good to Know

> **Good to know:** Assessment mode does not generate files. Use review-style prompts for validation and conversion-style prompts when you want artifacts written.

> **Good to know:** The skill asks for explicit output directories before writing plans, tests, and summaries.

> **Good to know:** Validation stops after two failed attempts rather than guessing at a third correction.

> **Good to know:** Generated tests depend on the shared `fixtures/` directory, so copy both the `.spec.ts` files and the fixtures when moving output into your Playwright project.

## Prerequisites

- Playwright installed in your project
- TypeScript support configured
- Acceptance criteria files in bullet markdown (`.md`) or Gherkin (`.feature`) format
- For validation: The skill's dependencies built (`npm run build` in the skill directory)

## What You Get

**Assessment mode output:**
- Validation report listing issues by file with specific problems, examples, and fixes
- Or confirmation that AC are conversion-ready with validated checklist

**Conversion mode output:**
- JSON test plan file (`<plans-dir>/<suite-slug>.json`)
- Playwright test suite (`<tests-output-dir>/<suite-slug>.spec.ts`)
- Summary metadata (JSON batch file in summary directory)
- Shared fixtures directory (error-handling.ts, console-tracking.ts)
- Optional Playwright config template

## Examples

### Example: Assessing AC Quality

```bash
# User provides AC file and asks:
"Review this feature file and tell me if it's ready for automation"

# Skill analyzes and reports issues:
❌ AC are not conversion-ready. Issues found:

File: login-flow.feature
1. Scenario "User logs in": Missing target intent
   - Problem: Test hook selector incomplete (button.form instead of button.form.submit)
   - Example: "clicks the button on the form"
   - Fix: Specify intent: "clicks the Submit button on the form"
```

### Example: Converting AC to Tests

```bash
# User provides AC file and specifies output directories:
"Convert form-validation.feature to Playwright tests. 
Put plans in ./test-plans, tests in ./e2e/tests, summaries in ./test-summaries"

# Skill generates:
# - ./test-plans/form-validation.json (validated plan)
# - ./e2e/tests/form-validation.spec.ts (Playwright suite)
# - ./test-summaries/batch-[timestamp].json (metadata)
# - ./e2e/tests/fixtures/ (shared utilities)
```

## Limits

- The skill plans the test structure but does not execute tests or verify they pass against your actual application
- Validation is limited to 2 attempts per plan (initial + 1 correction)
- Target selectors use controlled vocabulary; custom patterns require AC updates
- The skill cannot infer missing information—targets, values, and assertions must be explicit in AC
- For converting plans to tests, the skill relies on `scripts/translate-plan-to-tests.ts` which must be available

## Good to Know

> **Good to know:** AC that say "interact with" or "use" cannot be mapped cleanly to Playwright actions. Use specific verbs such as `click`, `fill`, or `select`.

> **Good to know:** Targets need the full `area.component.intent` pattern. A selector like `button.form` is too ambiguous; `button.form.submit` is automation-ready.

> **Good to know:** Test generation works best when AC include literal values like `'premium@example.com'` rather than descriptive placeholders such as "a valid email".

> **Good to know:** The skill only generates assertions for outcomes stated explicitly in the AC. If expected behavior is omitted, the resulting test will omit that assertion.
