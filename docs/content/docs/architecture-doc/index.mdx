## What It Helps You Do

Use this skill to generate or maintain an ARCHITECTURE.md document that gives engineers and agents an instant picture of how your system works. It discovers your tech stack, deployment model, data stores, and component relationships through codebase scanning and targeted questions.

The skill is especially useful when you need to:
- Create architecture documentation from scratch for a new or undocumented project
- Refresh stale architecture docs after infrastructure or stack changes
- Restructure free-form architecture notes into a standard template
- Document monorepo packages with proper cross-references to root-level docs

Activate this skill by saying "document my architecture", "create ARCHITECTURE.md", "update architecture", or mentioning "architecture overview" or "system architecture".

## When to Use

Use this skill when:
- Users say "architecture.md", "ARCHITECTURE.md", "document my architecture", "architecture overview", "system architecture"
- Users ask to "generate architecture doc", "create architecture file", "update architecture", or want "architecture diagram"
- Users want to "write up the system", "document the tech stack", or "create a technical overview"
- Working with undocumented codebases or codebases with drifted architecture docs

## How It Works

The skill works in three modes:

- **Create mode** scans the codebase, fills in what it can infer, and asks only about the gaps before drafting a new `ARCHITECTURE.md`.
- **Refresh mode** compares the existing document against current code and infrastructure signals, then updates only the sections that appear to have drifted.
- **Restructure mode** takes a nonstandard architecture document and maps it into the expected template while preserving the original content.

### Intelligent Discovery Process

Its discovery pass looks across project structure, stack, infrastructure, data/security, and testing so the interview stays focused on what could not be inferred automatically.

## Good to Know

> **Good to know:** The skill understands monorepo layouts. At the root, it generates system-wide docs covering all services. Inside a package, it creates focused docs that reference the root and avoid duplicating shared infrastructure details.

> **Good to know:** When refreshing docs, the skill reads existing content first and preserves all human-authored context. It never silently removes content — it surfaces changes and confirms whether they're still accurate.

> **Good to know:** For openspec-based projects, the skill checks for `openspec/config.yml` and uses it as the source of truth for stack facts and coding patterns, reducing redundant scanning work.

> **Good to know:** After writing ARCHITECTURE.md, the skill checks for AGENTS.md or CLAUDE.md and adds a reference to the architecture doc if one doesn't exist, helping agents understand system structure.

## Prerequisites

- Git repository (for hash-object commands during SHA computation)
- Standard project structure with recognizable config files (package.json, docker-compose.yml, etc.)
- Write permissions to create or modify ARCHITECTURE.md

## What You Get

A complete `ARCHITECTURE.md` draft covering project structure, system components, data stores, external services, deployment, security, development workflow, glossary, maintenance ownership, and future architecture notes.

Where the skill cannot infer enough, it leaves explicit `&lt;!-- TODO --&gt;` markers and shows a labeled preview before writing.

## Limits

- This skill generates documentation, it does not modify code or infrastructure
- It cannot infer subjective decisions (why you chose a particular database, why you structured components this way) — these require human input
- Section 2 (High-Level System Diagram) is generated as prose, not as visual diagrams or mermaid syntax
- Infrastructure details must be present in recognizable config files — the skill does not read cloud provider consoles or query live systems
