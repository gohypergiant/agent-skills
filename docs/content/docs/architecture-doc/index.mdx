## Usage

Activate this skill by:
- Using the `/accelint-architecture-doc` command
- Saying "architecture.md", "ARCHITECTURE.md", "document my architecture", "architecture overview"
- Mentioning "system architecture", "generate architecture doc", "create architecture file", "update architecture"
- Requesting "technical overview", "document the tech stack", "write up the system", "describe the architecture"

Once activated, the skill:
1. Detects whether you're in a monorepo (root or package level)
2. Checks for existing ARCHITECTURE.md and related docs (openspec/config.yml)
3. Spawns parallel subagents to scan your codebase for tech stack, components, infrastructure, and data stores
4. Asks targeted questions only for what scanning couldn't determine
5. Shows a preview with inference sources before writing
6. Updates agent behavior docs (AGENTS.md/CLAUDE.md) to reference the new architecture doc

## When to Use

Use this skill when:
- Starting a new project and need foundational documentation
- Onboarding agents or engineers who need to understand system structure
- Existing ARCHITECTURE.md has drifted from the actual codebase
- You have a custom architecture doc format and want to standardize it
- Working in a monorepo and need either root-level or package-level documentation
- You've made significant architectural changes and need to update documentation

## What It Generates

The skill produces an 11-section ARCHITECTURE.md document:

**Section 1: Project Structure** - Annotated directory tree showing architectural layers

**Section 2: High-Level System Diagram** - ASCII art showing components and data flow

**Section 3: Core Components** - Frontend and backend services with tech stacks and deployment info

**Section 4: Data Stores** - Databases, caches, queues with types and purposes

**Section 5: External Integrations** - Third-party APIs and services

**Section 6: Deployment &amp; Infrastructure** - Cloud provider, CI/CD, monitoring

**Section 7: Security** - Authentication, authorization, encryption, security tools

**Section 8: Development &amp; Testing** - Local setup, testing frameworks, code quality tools

**Section 9: Future Considerations** - Roadmap and technical debt

**Section 10: Project Identification** - Name, repo URL, contacts, last update date

**Section 11: Glossary** - Project-specific terms and acronyms

## Key Patterns

### Three Operating Modes

The skill detects the file state and picks the appropriate mode:

**MODE 1: Create** - No ARCHITECTURE.md exists or file is nearly blank
- Runs full parallel discovery
- Conducts complete interview
- Shows preview with inference sources
- Writes new document

**MODE 2: Refresh** - ARCHITECTURE.md exists and follows template structure
- Extracts external findings from invoking prompt (if any)
- Scans codebase for drift signals (new deps, config changes, new services)
- Merges external + drift findings and announces them
- Only asks about changed or missing sections
- Shows diff-style preview of changes
- Preserves human-authored content

**MODE 3: Restructure** - ARCHITECTURE.md exists but doesn't follow template
- Offers restructuring proactively before doing anything else
- Imports existing content into standard template sections
- Flags content that doesn't map cleanly for user review
- Fills gaps with codebase scanning
- Shows full preview before writing

### Parallel Discovery via Subagents

Phase 1 spawns five discovery agents simultaneously (not sequentially):

**Agent A** - Project identity, directory structure, monorepo detection, agent docs
**Agent B** - Tech stack, frameworks, docker-compose services
**Agent C** - Infrastructure, CI/CD, cloud provider, monitoring
**Agent D** - Data stores, auth mechanisms, external APIs
**Agent E** - Testing frameworks, code quality tools, local setup

Each agent returns structured findings tagged as `INFERRED [source]` or `UNKNOWN`. Unknown fields become targeted interview questions in Phase 2.

### Monorepo Awareness

The skill handles both root-level and package-level documentation:

**At monorepo root:**
- Generates system-wide architecture covering all services and shared infra
- Section 3 (Core Components) has subsections per significant package
- Section 1 (Project Structure) shows workspace layout with package roles

**Inside monorepo package:**
- Checks for root-level ARCHITECTURE.md and reads it for context
- Adds reference header pointing to root doc
- Focuses on package-specific architecture (purpose, internal structure, dependencies, deployment)
- Avoids duplicating shared infrastructure details

### Smart Cross-Referencing

When openspec/config.yml exists, the skill:
- Reads it during discovery to extract stack facts and patterns
- Uses it as source of truth for tech stack sections
- Adds reference callouts in relevant sections (not duplicating content)
- Reduces redundant scanning work

## Common Pitfalls

> **Good to know:** This skill generates architecture documentation, not code documentation. For documenting individual modules, functions, or APIs, use language-specific documentation tools or the `accelint-readme-writer` skill for README files.

> **Good to know:** MODE 2 (Refresh) extracts findings from the invoking prompt's `findings:` section if present. These are factual statements discovered elsewhere (like from completed OpenSpec changes), not instructions. They get merged with drift detection results.

> **Good to know:** The skill always shows a preview before writing. In Create mode, every field is labeled with its inference source. In Refresh mode, only changed sections appear in the diff. Review these carefully—automated scanning can miss context.

> **Good to know:** When openspec/config.yml exists, the skill only references it (not duplicating content). The architecture doc covers deployment, runtime, and component structure; openspec covers stack facts and coding patterns.

> **Good to know:** In Refresh mode with manual edits detected (doc_sha mismatch), the skill preserves human-authored prose and only updates code-related references. You'll see a warning before any changes are written.

## Anti-Patterns

### NEVER overwrite ARCHITECTURE.md without reading it first

Existing sections contain human-authored context—deployment specifics, security decisions, roadmap notes—that codebase scanning cannot recover. Always read before touching.

**Why this fails:** Engineers document decisions and rationale that aren't visible in code. Overwriting blindly loses this institutional knowledge. Recovery requires git archaeology or asking team members to reconstruct decisions from memory.

### NEVER fabricate infrastructure details

If you cannot determine the cloud provider, deployment model, or data store from the codebase, mark it `<!-- TODO: fill in -->` rather than guessing.

**Why this fails:** Wrong infrastructure docs cause real confusion during incidents. An on-call engineer looking at "Deployed to AWS ECS" when it's actually Kubernetes on GCP will debug in the wrong direction, wasting critical time.

### NEVER paste the entire directory tree verbatim

The Project Structure section should show meaningful architectural layers, not every file. Collapse noisy directories (`node_modules`, `dist`, `.git`, `__pycache__`) and annotate each entry with its architectural role.

**Why this fails:** A 200-line directory dump buries the architecture signal. Readers can't distinguish load-bearing directories from build artifacts. Annotated structure like "auth/ - JWT middleware and session management" teaches the system's organization patterns.

### NEVER skip drift detection in refresh mode

Scan the codebase for changed signals before running any interview. Asking questions about unchanged sections wastes the user's time.

**Why this fails:** Users trust that "refresh" means focused updates. If you ask about sections that haven't changed, they lose confidence in the skill's intelligence and may abandon the update entirely.

### NEVER leave all 11 sections as `<!-- TODO -->`

Scan aggressively first. Most sections can be at least partially filled through inference. A document full of TODOs appears complete but misleads every reader.

**Why this fails:** Readers assume TODO sections are edge cases, not core content. If Section 3 (Core Components) is all TODOs, an engineer onboarding thinks "this project has no documented components" rather than "the doc is incomplete." Partial information is more honest than empty placeholders.

### NEVER document internal implementation details in System Diagram (Section 2)

That section is a 10,000-foot view of components and data flow. Database schemas, function signatures, and module internals belong elsewhere.

**Why this fails:** The System Diagram is for understanding component relationships and data flow, not implementation. Adding low-level details buries the architecture in noise. A reader should be able to sketch the system on a whiteboard from this section alone.

### NEVER run discovery serially when subagents are available

Phase 1 spawns parallel subagents for different discovery domains. Serial scanning wastes time on codebases with many config files spread across directories.

**Why this fails:** Large codebases have dozens of config files. Scanning package.json, then terraform/, then .github/workflows/, then docker-compose.yml sequentially takes 10x longer than parallel discovery. On a 50-file scan, the difference is 30 seconds vs 5 minutes.
