## What It Helps You Do

Use this skill to create or update README.md files that accurately reflect your codebase. The skill analyzes your code structure, extracts public APIs, discovers usage examples from tests, and generates thorough documentation with practical, copy-pasteable examples.

It is especially useful when you need to:
- Create a new README for a project or package
- Update documentation after code changes
- Ensure README accuracy by comparing docs to actual exports
- Convert sparse documentation into comprehensive guides
- Generate human-sounding prose instead of robotic AI text

Activate this skill by:
- Saying "create a README", "update the README", or "document this package"
- Mentioning "README" in context of a specific directory or package
- Asking to fix outdated or incomplete documentation

## When to Use

Use this skill when:
- Creating a README.md for any project or package
- User asks to "document this" or "write a README"
- Existing README is stale after code changes
- Documentation needs auditing for completeness
- Working with monorepo packages that need individual READMEs

Do NOT use for:
- API documentation generation (use JSDoc/TSDoc tools instead)
- Changelogs or release notes
- Internal developer notes
- Non-Markdown documentation formats

## How It Works

The skill follows a systematic workflow to generate accurate, helpful documentation:

### Parallel Codebase Discovery

The skill spawns multiple discovery agents in parallel when available to efficiently analyze different aspects of your codebase:

- **Entry Points & Public API**: Traces `package.json` exports, main entry files, and all re-exported functions/types to map the complete public API
- **Dependencies & Configuration**: Identifies runtime dependencies, peer dependencies, package manager (from lockfiles), and available npm scripts
- **Examples & Usage**: Extracts real usage patterns from test files, `examples/` directories, and JSDoc `@example` blocks
- **Documentation Context**: Checks for existing README, CHANGELOG, CONTRIBUTING, and related docs

After discovery completes, the skill identifies documentation gaps such as missing exports, stale examples, outdated commands, or missing sections.

### Cross-Document Intelligence

Before scanning, the skill checks for related documentation:
- **openspec/config.yml**: Extracts package manager, tech stack, and key libraries to avoid redundant scanning
- **ARCHITECTURE.md**: References system architecture for cross-linking
- **AGENTS.md / CLAUDE.md**: Notes contribution guidelines

This ensures consistency (e.g., README uses the same package manager as config.yml) and creates proper cross-references.

### Gap Analysis and Updates

For existing READMEs, the skill compares discovered code structure against current documentation to identify:
- Public API exports not documented
- Code examples using deprecated patterns
- Wrong package manager commands
- Missing critical sections (installation, quick start, API reference)
- Signature mismatches between docs and actual code

### Human-Sounding Output

The skill generates natural-sounding prose by:
- Extracting real examples from tests and source code (never fabricating usage patterns)
- Using the `humanizer` skill to remove AI writing patterns like inflated language, promotional tone, and robotic phrasing
- Writing as if a helpful developer is explaining the package to a colleague

## Good to Know

> **Good to know:** This skill only documents public APIs accessible through package entry points. Internal helper functions not re-exported from `index.ts` are intentionally excluded.

> **Good to know:** Package manager detection is automatic based on lockfiles (`pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `bun.lockb`). All install commands use the detected manager.

> **Good to know:** The skill requires the `humanizer` skill to remove AI writing patterns. If not available, generated READMEs may contain robotic language.

> **Good to know:** In monorepos, README scope is determined by location. A README at `packages/my-lib/` documents only that package, not the entire repository.

## Prerequisites

- A codebase with identifiable entry points (`package.json` with `main`, `module`, `types`, or `exports` fields)
- The `humanizer` skill must be available for natural-sounding output

## What You Get

A complete README.md file with:
- Installation instructions using the correct package manager
- Quick start examples extracted from real code
- API reference documenting all public exports
- Usage examples from test files or `examples/` directories
- Cross-references to ARCHITECTURE.md, AGENTS.md, or other related docs (when they exist)
- Human-sounding prose free of AI writing patterns

For updates to existing READMEs, you receive:
- A list of identified gaps (missing exports, stale examples, outdated commands)
- Suggested changes before applying updates
- Preserved structure with targeted fixes to outdated sections
