## Usage

Activate this skill by:
- Using the `/accelint-readme-writer` command
- Saying "create a README", "update the README", "document this package"
- Mentioning "README.md" while working on documentation
- Working on documentation in project or package directories

```bash
# For a specific directory
/accelint-readme-writer packages/my-lib

# Current working directory
/accelint-readme-writer

# Update existing README
/accelint-readme-writer .
```

Once activated, the skill analyzes your codebase (entry points, exports, dependencies, examples, tests) and either generates a new README from scratch or suggests specific updates to bring existing documentation in sync with the code.

## When to Use

Use this skill when:
- Creating a new README.md for a project or package
- Updating an existing README.md after code changes
- Auditing documentation for completeness and accuracy
- Converting sparse documentation into thorough guides
- User asks to "document this package" or "write a README"
- User mentions README in context of a monorepo subdirectory

## What It Does

**Codebase Analysis** - Entry points and public API mapping, dependency and configuration detection, example extraction from tests and JSDoc, documentation context review

**Gap Detection** - Missing exports not documented, stale examples using deprecated patterns, missing sections (installation, quick start, API reference), outdated commands (wrong package manager, missing scripts)

**Documentation Generation** - Follows standard README structure (heading, installation, quick start, what/why, API, examples, license), extracts real examples from test files and JSDoc blocks, detects correct package manager from lockfiles, writes human-sounding prose (uses humanizer skill to remove AI patterns)

**Smart Updates** - Compares code against existing README, identifies what's missing and what's stale, merges external findings (from OpenSpec changes, etc.), preserves manual edits while updating code-related sections

## Examples

### Example: Creating README for New Package

```bash
/accelint-readme-writer packages/config-parser
```

The skill will:
1. Check for related docs (openspec/config.yml, ARCHITECTURE.md, AGENTS.md)
2. Spawn parallel discovery agents to analyze:
   - Entry points and public API
   - Dependencies and configuration
   - Examples and usage patterns
   - Existing documentation context
3. Generate a complete README following the standard structure
4. Run humanizer skill to remove AI writing patterns
5. Write the README to `packages/config-parser/README.md`

### Example: Updating Existing README

```bash
/accelint-readme-writer .
```

Output shows gaps:
```
Found gaps in README:
- Missing exports: validateConfig, ConfigSchema
- Stale example: parse() signature changed (added options param)
- Wrong package manager: using npm but lockfile is pnpm-lock.yaml
- Missing section: API Reference
```

After user confirmation, updates the README preserving the existing structure and manual edits.

### Example: README with External Findings

When invoked from an OpenSpec change context:
```bash
# User provides findings from completed change
/accelint-readme-writer

findings:
- Added new parseYaml export in src/index.ts
- Deprecated parseConfig in favor of parse
- Added streaming mode option to ParseOptions
```

The skill merges these findings with codebase scan results and updates the README accordingly.

## Common Pitfalls

> **Good to know:** This skill uses parallel sub-agents when available to speed up codebase analysis. If sub-agents aren't available, it performs the same analysis inline but sequentially.

> **Good to know:** The skill requires the `humanizer` skill to remove AI writing patterns from generated content. If humanizer isn't available, you'll see a warning and should enable it in settings.

> **Good to know:** In monorepos, the README scope is determined by the directory you specify. A README at the repo root documents the entire monorepo, while a README in `packages/my-lib` documents only that package.

> **Good to know:** The skill reads from `skills/` directory only, ignoring `.agents/skills`, `.claude/skills`, and `.pi/skills` to avoid duplicate analysis of symlinked skills.

## Anti-Patterns

**NEVER run discovery serially when sub-agents are available** — spawn parallel discovery agents for different aspects (entry points, dependencies, examples, existing docs) to analyze the codebase efficiently. Serial file-by-file scanning wastes time.

**NEVER document non-exported internal functions** — only document the public API that's accessible through package entry points. Internal helper functions that aren't re-exported from `index.ts` don't belong in the README.

**NEVER fabricate usage examples** — extract real examples from test files, JSDoc blocks, or `examples/` directories. Made-up examples often contain subtle errors that confuse users.

**NEVER use the wrong package manager commands** — check for lockfiles (`pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `bun.lockb`) and use the matching package manager in all commands. Wrong commands break the user's first experience.

**NEVER skip comparing code to existing README** — when updating documentation, identify what's missing, what's stale, and what signature changes occurred. Silent drift between code and docs causes user frustration.

**NEVER write robotic, AI-sounding text** — use the humanizer skill to remove inflated language, promotional tone, and AI writing patterns. Documentation should sound like a helpful human wrote it.
