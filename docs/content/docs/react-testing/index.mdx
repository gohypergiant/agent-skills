## What It Helps You Do

Use this skill to write React component tests that verify user-facing behavior rather than implementation details. It guides you toward accessible, maintainable tests that catch real bugs while remaining resilient to refactoring.

Apply this skill when:
- Writing or reviewing tests with Testing Library
- Choosing the right query method (getByRole vs getByTestId)
- Simulating user interactions (clicks, typing, form submission)
- Testing async behavior (data loading, timeouts, promises)
- Setting up components that need providers (Context, Redux, Router)
- Improving test reliability and reducing flakiness

## When to Use

Activate this skill when you encounter:
- `render()`, `screen`, `fireEvent`, `userEvent`, or `waitFor` in test files
- `*.test.tsx` or `*.test.jsx` files with React components
- Testing Library queries: `getByRole`, `findBy*`, `queryBy*`
- Test setup requiring React context or state management
- Keywords: RTL, Testing Library, jest-dom, toBeInTheDocument

## How It Works

The skill uses query difficulty as a quality signal—if queries are hard, your component likely has accessibility problems. It teaches you to:

**Query by user behavior first**: Start with `getByRole` (how screen readers find elements), fall back to `getByLabelText` (how sighted users read forms), and use `getByTestId` only as a last resort when accessibility is impossible.

**Simulate realistic interactions**: Use `userEvent.click()` instead of `fireEvent.click()` because real users trigger focus → mousedown → mouseup → click, not just a single click event. Components that work with fireEvent sometimes break in production.

**Test outcomes, not internals**: Assert what users see and do (rendered text, interaction results), not how your component achieves it (state variables, function calls). This keeps tests resilient to refactoring while catching actual UX bugs.

**Handle async correctly**: Use `findBy*` queries for elements that load asynchronously instead of `waitFor` loops. Use `waitForElementToBeRemoved` for disappearance. Each async utility has specific semantics that prevent flaky tests.

**Set up providers once**: Create custom render utilities that wrap components in required providers (Context, Redux, Router) automatically instead of repeating setup in every test.

## Examples

### Example: Query Selection

```tsx
// ❌ Bad: Test ID provides no accessibility confidence
const button = screen.getByTestId('submit-button');

// ✅ Good: Role query verifies button is accessible
const button = screen.getByRole('button', { name: /submit/i });
```

### Example: User Interactions

```tsx
// ❌ Bad: fireEvent misses the full interaction sequence
fireEvent.click(button);

// ✅ Good: userEvent simulates realistic user behavior
await userEvent.click(button);
```

### Example: Async Content

```tsx
// ❌ Bad: Polling with waitFor when findBy solves it directly
await waitFor(() => expect(screen.getByText('loaded')).toBeInTheDocument());

// ✅ Good: findBy waits for element to appear
expect(await screen.findByText('loaded')).toBeInTheDocument();
```

### Example: Custom Render with Providers

```tsx
// ❌ Bad: Repeating provider setup in every test
render(
  <ThemeProvider>
    <UserContext.Provider value={mockUser}>
      <Component />
    </UserContext.Provider>
  </ThemeProvider>
);

// ✅ Good: Centralized custom render utility
function renderWithProviders(ui, { user = mockUser, ...options } = {}) {
  return render(ui, {
    wrapper: ({ children }) => (
      <ThemeProvider>
        <UserContext.Provider value={user}>{children}</UserContext.Provider>
      </ThemeProvider>
    ),
    ...options,
  });
}

renderWithProviders(<Component />);
```

## Good to Know

> **Good to know:** Higher queries in the hierarchy provide more accessibility confidence. If you can't use `getByRole` or `getByLabelText`, that's a sign your component needs accessibility improvements before it ships.

> **Good to know:** Use `screen.debug()` when queries fail to see the current DOM structure, or `screen.logTestingPlaygroundURL()` for an interactive tool showing what queries work for your rendered output.

> **Good to know:** `queryBy*` returns null silently—use `getBy*` for better error messages when an element should exist. Reserve `queryBy*` only for asserting absence with `.not.toBeInTheDocument()`.

> **Good to know:** Act warnings usually mean promises resolved after test completion or you forgot `await` on async queries. They're not caused by correct use of `findBy` or `waitFor`.

## Prerequisites

- `@testing-library/react` installed
- `@testing-library/user-event` for realistic interactions
- `@testing-library/jest-dom` for extended matchers (toBeInTheDocument, toBeVisible, etc.)
- Test runner: vitest or jest

## What You Get

When auditing existing tests (via `/accelint-react-testing <path>`), you receive a structured report with:
- Severity-ranked issues (Critical → Low)
- Impact analysis (accessibility confidence, reliability, refactor safety)
- Specific line numbers and file paths
- Pattern references for detailed guidance
- Summary metrics for tracking improvements

For direct questions or writing new tests, you get immediate guidance on the right patterns without formal reporting overhead.
