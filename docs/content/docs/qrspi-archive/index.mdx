## What It Helps You Do

Use this skill to archive OpenSpec changes completely—not just the merge, but also the cross-capability relationship graph and index bookkeeping that OpenSpec doesn't maintain automatically.

This skill invokes `/opsx:archive` or `/opsx:bulk-archive` itself as its first phase, waits for the merge to fully resolve, then:
- Links every pair of capabilities the change touched via a shared `related:` frontmatter list
- Keeps `openspec/specs/INDEX.md` current by patching only the rows that changed
- Appends a row to the append-only changelog at `openspec/changes/archive/INDEX.md`

Cross-linking happens after the merge resolves, not before—a delta spec in `openspec/changes/<slug>/specs/` is provisional until conflicts are resolved. Only the merged, archived spec is worth indexing.

Activate this skill by:
- Using the `/accelint-qrspi-archive` command
- Saying "archive this change", "bulk archive these changes", "run opsx:archive", "update the specs index", or "cross-link the specs"

## When to Use

Use this skill when:
- You want to archive a change and ensure its capabilities are properly cross-linked
- Users say "archive this change", "bulk archive these changes", "run opsx:archive", "run opsx:bulk-archive"
- You need to update the specs index or cross-link specs after a merge
- You want the archived-change changelog kept current

## How It Works

The skill handles archiving in four steps:

1. **Preflight** — checks that the change has the metadata and spec structure needed for archiving
2. **Archive** — runs `/opsx:archive` or `/opsx:bulk-archive` and waits for the merge to finish
3. **Cross-link and index** — updates `related:` links across touched specs and patches the affected rows in `openspec/specs/INDEX.md`
4. **Append changelog** — adds one row per archived change to `openspec/changes/archive/INDEX.md`

The linking side is additive only: it updates relationships and archive bookkeeping, but does not prune stale links or reclassify prior change status.

## Prerequisites

- OpenSpec CLI installed and initialized
- Sub-agent support for per-capability spec writes (normal operation requires this; see error handling for the degraded fallback)
- Each change's `openspec/changes/<slug>/design.md` should have `specs_touched` and `decisions` frontmatter (the skill can derive these if missing, but requires your confirmation)
- Every touched spec must have a `## Purpose` heading in its body

## What You Get

- The change(s) archived via OpenSpec's native merge
- Updated `related:` frontmatter and regenerated `## Related Specs` sections in every touched spec
- A patched `openspec/specs/INDEX.md` with only the touched rows changed
- One appended row per archived change in `openspec/changes/archive/INDEX.md`
- A summary report showing what changed and what to review before committing

## Examples

### Example: Archiving a single change

```bash
/accelint-qrspi-archive
```

The skill runs preflight checks, archives the change through OpenSpec, updates touched spec relationships, patches the relevant specs index rows, and appends the archive changelog.

### Example: Archiving with missing frontmatter

```bash
/accelint-qrspi-archive
```

If required archive metadata is missing, the skill derives a candidate from the change files and asks you to confirm or edit it before continuing.

## Good to Know

> **Good to know:** This skill is the entry point for archiving. It runs the native OpenSpec archive command itself rather than assuming someone already merged the change.

> **Good to know:** The archive step stays interactive, so prompts raised during archiving reach you directly.

> **Good to know:** The linking side is additive only. Removing stale `related:` entries or changing archive status belongs to `accelint-archive-synthesis`.

## Limits

- This skill does not prune `related:` entries or change existing `Status` values—that requires cross-change judgment and human confirmation
- It does not verify `specs/INDEX.md` against every `spec.md` on disk (corpus-wide drift detection is `accelint-archive-synthesis`'s domain)
- It does not reconcile summary stats trailing `changes/archive/INDEX.md` (e.g., total row counts)
- It does not model relationship directionality—co-touch is inherently symmetric

## Related
