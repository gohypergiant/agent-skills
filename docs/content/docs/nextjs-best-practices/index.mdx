## What It Helps You Do

Use this skill to optimize Next.js applications for performance and security. It helps you:

- Eliminate server-side data fetching waterfalls
- Secure Server Actions with proper authentication and validation
- Reduce RSC serialization overhead and response sizes
- Implement parallel data fetching patterns
- Choose correctly between Server and Client Components
- Place Suspense boundaries strategically
- Fix authentication issues in Server Actions

The skill catches common Next.js anti-patterns during code review and suggests optimized alternatives with performance impact metrics.

## When to Use

Use this skill when:

- Writing or refactoring Server Components, Client Components, or Server Actions
- Users say "optimize this Next.js code", "this page is slow", "secure this Server Action"
- Implementing data fetching patterns with `fetch()`, `React.cache()`, or database queries
- Reviewing Next.js code for performance bottlenecks
- Working with App Router or Pages Router
- Debugging RSC serialization duplication or large HTML payloads
- Deciding between server and client component boundaries

## How It Works

The skill focuses on three areas:

### Core Optimization Areas

**Server-side performance:**
- Prevents waterfall chains by starting operations immediately
- Parallelizes independent data fetches with `Promise.all()` or `Promise.allSettled()`
- Deduplicates requests using `React.cache()`
- Minimizes RSC serialization by passing only necessary data
- Uses `after()` for non-blocking background operations

**Security:**
- Validates and authenticates every Server Action
- Prevents data exposure through proper input validation
- Implements authorization checks before database operations
- Avoids exposing sensitive data through serialization

**Component architecture:**
- Places Suspense boundaries strategically to unblock rendering
- Decides server vs client component placement based on interactivity needs
- Avoids barrel imports that slow development builds
- Structures components to enable parallel data streaming


## Examples

### Example: Eliminating Data Fetching Waterfalls

```tsx
// Before: Sequential fetching (waterfall)
async function Page() {
  const user = await getUser();
  const posts = await getPosts(user.id);
  return <div>...</div>;
}

// After: Parallel fetching
async function Page() {
  const userPromise = getUser();
  const user = await userPromise;
  const postsPromise = getPosts(user.id);
  const posts = await postsPromise;
  return <div>...</div>;
}
```

### Example: Securing Server Actions

```tsx
// Before: No authentication
export async function deletePost(postId: string) {
  await db.delete(postId);
}

// After: Authenticated and validated
export async function deletePost(postId: string) {
  const session = await auth();
  if (!session?.user) throw new Error('Unauthorized');
  
  const post = await db.getPost(postId);
  if (post.authorId !== session.user.id) {
    throw new Error('Forbidden');
  }
  
  await db.delete(postId);
}
```

### Example: Minimizing RSC Serialization

```tsx
// Before: Over-serialization (entire user object)
async function UserProfile({ userId }: Props) {
  const user = await db.users.findUnique({ where: { id: userId } });
  return <Profile user={user} />;
}

// After: Minimal serialization (only needed fields)
async function UserProfile({ userId }: Props) {
  const user = await db.users.findUnique({
    where: { id: userId },
    select: { name: true, avatar: true, bio: true }
  });
  return <Profile user={user} />;
}
```
