## What It Helps You Do

Use this skill to implement OpenSpec changes that were planned using QRSPI methodology. It automates the implementation phase by analyzing task dependencies, spawning parallel sub-agents where possible, and managing the complete workflow from execution through verification.

The skill is especially useful when you need to:
- Apply a QRSPI-planned change with independent task slices
- Speed up implementation by running parallel work streams
- Ensure living documents stay synchronized with code changes
- Verify implementation completeness before archival

Activate this skill by:
- Using the `/accelint-qrspi-apply` command
- Saying "apply this QRSPI change" or "implement with parallelization"
- Mentioning "run the parallel slices" or "apply change with vertical slices"

## Prerequisites

- **OpenSpec CLI** installed and initialized
- **QRSPI-planned change** created via `accelint-qrspi-propose` skill
  - Must include "Parallelization Strategy" section in tasks.md
  - Tasks must use markdown checklist format (`- [ ] task`)
- **Sub-agent support** for parallel execution
- **Git repository** initialized

> **Good to know:** This skill is specifically designed for QRSPI changes with parallelization strategies. Standard OpenSpec changes should use `/opsx:apply` directly.

## When to Use

Use this skill when:
- The user wants to "apply a QRSPI change"
- Working with changes that include task slicing and parallelization strategies
- Implementing changes where independent slices can run simultaneously
- You need intelligent orchestration beyond simple sequential task execution

## How It Works

The skill follows a staged implementation workflow:

### 1. Validate and Plan Execution

It selects the target change, confirms that `tasks.md` is present, and reads the parallelization strategy to determine which slices can run together and which must wait on dependencies.

### 2. Run Slices in Order

Independent slices run in parallel, while dependent slices wait for earlier work to finish. Progress is tracked through the task checkboxes in `tasks.md`, which also lets the skill resume after a pause or context clear.

### 3. Refresh Living Docs

Before verification, the skill updates the project documents affected by the change, including OpenSpec config, architecture docs, agent instructions, and README content where relevant.

### 4. Verify Readiness

It finishes with `/opsx:verify`, checking that tasks are complete, requirements are implemented, and documentation stays in sync.

## What You Get

After successful completion:
- **Fully implemented change** with all tasks complete
- **Updated living documents** synchronized with code changes
- **Verification report** showing status, issues (if any), and next steps
- **Clear archive guidance** if verification passed

## Examples

### Example: Applying a parallelized change

```bash
/accelint-qrspi-apply remove-security-ruleset
```

The skill analyzes slice dependencies, runs independent work in parallel, updates affected docs, and ends with a verification report.

### Example: Resuming after a pause

```bash
/accelint-qrspi-apply auth-refactor
```

If work was already partially completed, the skill reads the task checkboxes and resumes from the next incomplete slice.

## Good to Know

> **Good to know:** If no parallelization strategy exists, the skill falls back to sequential execution.

> **Good to know:** Tasks must use markdown checklist format (`- [ ] task`) so progress can be tracked and resumed.

> **Good to know:** Documentation updates happen before verification so the final check can catch stale docs.

> **Good to know:** Long implementations can pause between dependency levels and resume later without losing progress.

## Limits

- **Planning only via QRSPI**: This skill applies changes planned with `accelint-qrspi-propose`. It does not create plans or modify specs.
- **No automatic archival**: After verification passes, the skill suggests archival but does not automatically archive. Use `/accelint-qrspi-archive` to complete the workflow.
- **Requires sub-agent support**: Parallel execution depends on sub-agent capabilities. Without sub-agent support, the skill falls back to sequential implementation.
- **Checklist format required**: Tasks must use `- [ ]` format for progress tracking. Other formats (numbered lists, plain bullets) are not supported.
